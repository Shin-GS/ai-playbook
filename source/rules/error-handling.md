# 에러 처리 전략

## 1. 에러 분류 체계

에러는 **복구 가능 여부**와 **원인 주체** 두 축으로 분류한다.

| 분류 | 설명 | 예시 | 대응 |
|------|------|------|------|
| Client Error | 사용자 입력/요청 오류 | 잘못된 파라미터, 인증 실패 | 안내 메시지 반환, 재시도 불필요 |
| Business Error | 비즈니스 규칙 위반 | 잔액 부족, 재고 없음 | 사용자에게 원인 안내 |
| Transient Error | 일시적 외부 장애 | 타임아웃, 네트워크 끊김, 503 | 재시도 대상 |
| System Error | 내부 버그 또는 치명적 장애 | NPE, DB 커넥션 풀 고갈 | 알림 + 즉시 조사 |

> **왜**: 모든 에러를 동일하게 처리하면 재시도할 필요 없는 에러를 반복하거나, 재시도해야 할 에러를 무시하게 된다.

## 2. 사용자 메시지 vs 개발자 메시지

```
┌─────────────────────────────────────────────────┐
│  사용자 메시지 (userMessage)                      │
│  - 무엇이 잘못되었는지 + 무엇을 하면 되는지        │
│  - 내부 구현 세부사항 미포함                       │
│  - 다국어 지원 가능한 코드/키 기반 권장            │
├─────────────────────────────────────────────────┤
│  개발자 메시지 (devMessage / log)                 │
│  - 스택 트레이스, 파라미터 값, 호출 경로           │
│  - 재현에 필요한 모든 컨텍스트                     │
│  - 민감 정보 마스킹 (토큰, 비밀번호)              │
└─────────────────────────────────────────────────┘
```

규칙:
1. 응답 바디에는 `userMessage`(또는 error code)만 포함한다.
2. `devMessage`는 로그에만 기록한다.
3. 에러 응답에 trace ID를 포함하여 사용자 문의 시 로그 추적을 가능하게 한다.
4. 프로덕션에서 스택 트레이스를 클라이언트에 노출하지 않는다.

> **왜**: 사용자에게 기술적 세부사항을 보여주면 혼란만 주고, 공격자에게 내부 구조 힌트를 제공한다.

## 3. 에러 응답 포맷

```json
{
  "error": {
    "code": "INSUFFICIENT_BALANCE",
    "message": "잔액이 부족합니다. 충전 후 다시 시도해주세요.",
    "traceId": "abc-123-def"
  }
}
```

- `code`: 머신이 읽을 수 있는 고정 문자열 (클라이언트 분기용)
- `message`: 사람이 읽는 안내 메시지
- `traceId`: 로그 추적용 고유 ID

## 4. 로깅 기준

| 레벨 | 사용 기준 | 예시 |
|------|-----------|------|
| ERROR | 즉시 대응 필요, 시스템 영향 | DB 연결 실패, 결제 처리 실패 |
| WARN | 정상은 아니지만 서비스 지속 가능 | 재시도 성공, 폴백 동작 |
| INFO | 비즈니스 흐름 추적 | 주문 생성, 사용자 로그인 |
| DEBUG | 개발 시 디버깅 | 파라미터 값, SQL 쿼리 |

규칙:
1. ERROR 로그에는 반드시 **컨텍스트**(요청 ID, 사용자 ID, 입력 파라미터)를 포함한다.
2. catch 블록에서 에러를 삼키지(swallow) 않는다 — 최소 WARN 이상으로 기록한다.
3. 민감 정보(비밀번호, 카드번호, 토큰)는 마스킹 후 로깅한다.
4. 반복 에러는 동일 메시지의 빈도를 제한한다 (log flooding 방지).

> **왜**: 에러를 삼키면 장애 원인을 추적할 수 없고, 과도한 로깅은 저장 비용과 노이즈를 유발한다.

## 5. 재시도 전략

### 재시도 대상 (Transient Error만)

- 네트워크 타임아웃 / 연결 거부
- HTTP 503, 429
- 데드락으로 인한 트랜잭션 실패

### 재시도 금지 대상

- 4xx 클라이언트 에러 (400, 401, 403, 404)
- 비즈니스 규칙 위반
- 데이터 무결성 에러 (unique constraint 등)

### 재시도 파라미터

| 파라미터 | 권장 기본값 | 설명 |
|----------|-------------|------|
| maxRetries | 3 | 최대 재시도 횟수 |
| initialDelay | 500ms | 첫 재시도 대기 시간 |
| backoffMultiplier | 2 | 지수 백오프 배수 |
| maxDelay | 10s | 대기 시간 상한 |
| jitter | ±20% | 동시 재시도 회피 (thundering herd 방지) |

### 재시도 패턴

```
retry(maxRetries=3, backoff=exponential) {
  1차 시도 → 실패 → 500ms 대기
  2차 시도 → 실패 → 1000ms + jitter 대기
  3차 시도 → 실패 → 2000ms + jitter 대기
  → 최종 실패: 에러 전파 + 알림
}
```

규칙:
1. 재시도는 **멱등성이 보장된 작업**에만 적용한다.
2. 비멱등 작업(결제, 발송)은 재시도 전 중복 체크 로직을 먼저 구현한다.
3. 재시도 횟수를 넘기면 Circuit Breaker 패턴을 고려한다.
4. 재시도 시 매번 로그를 남긴다 (WARN 레벨).

> **왜**: 무조건 재시도하면 장애가 전파(cascade)되고, 멱등성 없이 재시도하면 중복 처리가 발생한다.

## 6. 예외 처리 레이어 규칙

1. **Controller/Handler**: 글로벌 에러 핸들러로 일괄 처리. 개별 try-catch 최소화.
2. **Service**: 비즈니스 예외를 명시적으로 throw. catch는 복구 가능한 경우에만.
3. **Infrastructure**: 외부 시스템 예외를 도메인 예외로 변환하여 상위에 전파.
4. **Client (HTTP/gRPC)**: 타임아웃과 재시도를 클라이언트 레이어에서 처리.

> **왜**: 예외 처리가 산재하면 일관성이 깨지고 에러 흐름을 추적할 수 없다.

## 7. 피해야 할 패턴

- `catch (e) {}` — 에러를 삼키고 아무것도 안 함
- 모든 에러에 동일한 "서버 에러입니다" 메시지 반환
- 재시도 로직에 지수 백오프 없이 즉시 재시도 (서버 부하 가중)
- checked exception을 무의미하게 감싸서 다시 throw (정보 손실)
- 에러 로그에 컨텍스트 없이 메시지만 기록
- HTTP 200으로 에러를 반환 (클라이언트가 에러를 감지 못함)
