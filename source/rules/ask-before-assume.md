# 되묻기 규칙

## 목적

AI가 불확실한 상황에서 멋대로 추측하여 구현하면, 사용자가 나중에 발견했을 때 되돌리는 비용이 크다. "기능 인식 부채"를 줄이기 위해, 확인이 필요한 상황에서는 반드시 사용자에게 되묻는다.

## 반드시 확인할 것 (진행 전 사용자 동의 필요)

- 요구사항에서 **2가지 이상 해석이 가능**할 때
- 기존 코드/기획과 **충돌하는 변경**일 때
- **삭제나 대체**가 포함될 때 (되돌리기 어려운 변경)
- **새 의존성**(라이브러리, 패키지) 추가가 필요할 때
- **아키텍처 변경** (새 레이어, 패턴 도입, 구조 변경)
- 사용자가 명시하지 않은 **추가 기능**을 넣으려 할 때
- **비용이 발생**하는 선택 (유료 API, 인프라 추가)
- **보안에 영향**을 미치는 변경 (인증 방식 변경, 권한 수정)

## 확인 없이 진행할 것 (AI 자율 판단 허용)

- 네이밍, 포맷팅 같은 스타일 결정 (기존 패턴 따름)
- 기존 패턴을 따르는 구현 (명확한 전례가 있으면)
- 에러 핸들링 추가 (안전한 방향의 보강)
- 기존 요구사항에 명시된 내용의 구현
- 누락된 import/타입 추가
- 린트/빌드 에러 수정 (명백한 수정)
- 문서/주석 업데이트

## 되묻는 방식

- 짧고 명확하게: "A와 B 중 어떤 방향으로 할까요?"
- 선택지를 제시: 각 선택의 트레이드오프를 한 줄로
- 추천을 포함: "저는 A를 추천합니다. 이유: ..."
- 한번에 여러 질문 묶지 않기 (1~2개씩)

## 안티패턴

- "일단 이렇게 하겠습니다" → 확인 후 진행해야 할 상황에서 단독 결정
- "나중에 수정하면 됩니다" → 되돌리기 비용을 과소평가
- 매 줄마다 물어보기 → 위 "확인 없이 진행" 기준에 해당하면 자율 진행
