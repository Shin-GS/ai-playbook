# 테스트 전략

## 목적

"뭘 자동 테스트로, 뭘 수동 QA로 확인할지" 명확한 기준을 제공한다.
둘 다 하면 좋지만, 1인 빌더 환경에서 효율적으로 배분하는 것이 핵심.

## 자동 테스트 대상 (코드로 작성)

| 대상 | 이유 | 테스트 종류 |
|------|------|------------|
| 비즈니스 로직 계산 | 수식 정확성 필수 | 단위 테스트 |
| 상태 전이 | 조건별 분기 검증 | 단위 테스트 |
| API 응답 형식/상태코드 | 계약 보장 | 통합 테스트 |
| 입력 검증 (유효/무효) | 경계값 커버 | 단위 테스트 |
| DB 쿼리 결과 | 데이터 정합성 | 통합 테스트 |
| 인증/인가 | 보안 필수 | 통합 테스트 |

## 수동 QA 대상 (체크리스트로 확인)

| 대상 | 이유 | 확인 방법 |
|------|------|-----------|
| UI 레이아웃/시각적 확인 | 자동화 비용 대비 효과 낮음 | 브라우저에서 눈으로 |
| 사용자 흐름 (여러 화면 걸치는 시나리오) | 통합 복잡도 높음 | 시나리오 따라 조작 |
| 외부 시스템 연동 (결제, 알림) | mock으로 커버 안 되는 실제 동작 | 실제 환경 테스트 |
| 모바일 실기기 확인 | 에뮬레이터 한계 | 실기기 조작 |
| 반응형 레이아웃 | 다양한 사이즈 조합 | 브라우저 리사이즈 |
| 접근성 (키보드 네비, 스크린 리더) | 도구+수동 병행 | 키보드/도구 확인 |

## 둘 다 하면 좋은 것 (여유 있을 때)

- E2E 테스트 (핵심 사용자 흐름 자동화 — Playwright/Cypress)
- 시각적 회귀 테스트 (스크린샷 비교)
- 성능 테스트 (응답 시간 측정)

## 우선순위 판단 기준 (1인 빌더용)

| 조건 | 자동 테스트 | 수동 QA |
|------|------------|---------|
| 돈이 오가는 기능 | ✅ 필수 | ✅ 필수 |
| 핵심 사용자 흐름 | ✅ 권장 | ✅ 필수 |
| 일반 CRUD | ⚡ 있으면 좋음 | ✅ 기능 변경 시 |
| 보조 기능 | ❌ 생략 가능 | ⚡ 대규모 변경 시만 |

## 테스트 작성 시점

| 시점 | 행동 |
|------|------|
| 새 기능 구현 완료 시 | 자동 테스트: 핵심 로직 단위 테스트 작성. QA: 체크리스트 생성. |
| 버그 수정 완료 시 | 자동 테스트: 해당 버그 재현 테스트 추가 (회귀 방지). QA: regression 마킹. |
| 리팩토링 완료 시 | 자동 테스트: 기존 테스트 통과 확인만. QA: 불필요. |
| 비즈니스 규칙 변경 시 | 자동 테스트: 관련 단위 테스트 수정. QA: 영향받는 체크리스트 갱신. |

## verification-loop과의 관계

`verification-loop` workflow의 "테스트" 단계에서:
- **자동 테스트**: 빌드/린트와 함께 자동 실행
- **수동 QA**: verification-loop에 포함하지 않음. 배포 전 별도 수행.
