# 작업 도구 선택 전략

## 기본 원칙

> **기본값은 "main이 직접, 도구 없이" 처리.**
> 특정 조건을 만족할 때만 도구를 활성화한다.
> 도구는 공짜가 아니다 — 각각 비용(컨텍스트 소비, 관리 복잡도, 시간)이 있으므로 이점이 비용을 초과할 때만 사용한다.

---

## 도구 카탈로그

### 병렬 실행 도구

| 도구 | 용도 | 비용 | 가치 있는 상황 |
|------|------|------|--------------|
| 서브에이전트 (병렬) | 같은 코드베이스에서 역할 분리 (FE/BE 동시 등) | 프롬프트 전달 + 결과 수신 + 컨텍스트 소비 | 대량 코드 수정, FE+BE 동시 변경 |
| 서브에이전트 (위임) | 전문 역할에 맡기기 (리뷰, 기획, 디자인) | 동일 | 역할별 전문성 필요, 결과물 형식이 정해진 작업 |
| git worktree | 독립적인 2개 작업을 물리적으로 분리 | 생성/정리 + 작업 전환 + 머지 충돌 가능성 | 30분+ 걸리는 독립 작업 병행, 핫픽스 + 개발 동시 |

### 컨텍스트 관리 도구

| 도구 | 용도 | 비용 | 가치 있는 상황 |
|------|------|------|--------------|
| context-gatherer | 넓은 범위 코드 구조 탐색 | 서브에이전트 1회 호출 비용 | 처음 보는 코드베이스, 10파일+ 탐색 필요 |
| MCP load_asset | playbook 도메인 지식 참조 | MCP 1회 호출 | 특정 도메인 지식 필요 (2단계) |

### 품질 향상 도구

| 도구 | 용도 | 비용 | 가치 있는 상황 |
|------|------|------|--------------|
| verification-loop | 코드 변경 후 검증 사이클 | 빌드/테스트 실행 시간 | 코드 변경 시 항상 (필수) |
| multi-perspective-review | 의사결정 품질 향상 (4관점) | 분석 시간 + 컨텍스트 소비 | 기획/디자인/아키텍처 결정 시 |
| 코드 리뷰어 (서브에이전트) | 변경 코드 품질 검증 | 서브에이전트 1회 | 코드 변경량이 클 때 |

## 판단 플로우차트

```
작업 요청 접수
  │
  ├─ 단순 수정인가? (1파일, 5줄 이내, 패턴 명확)
  │   └─ YES → main 직접 처리. 도구 없음.
  │
  ├─ 코드 변경인가?
  │   ├─ FE + BE 동시? → 서브에이전트 병렬 호출
  │   ├─ 전문 역할 필요? (리뷰, 아키텍처) → 해당 서브에이전트 위임
  │   ├─ 대량 수정 (10파일+)? → 서브에이전트 위임 (main 컨텍스트 보호)
  │   └─ 단일 영역, 소량 → main 직접
  │   └─ 변경 완료 → verification-loop (항상)
  │
  ├─ 현재 작업 중인데 다른 작업 요청?
  │   ├─ 독립적 + 30분+ 예상? → git worktree 분리 제안
  │   ├─ 5분 이내 끝남? → stash → 빠르게 처리 → pop
  │   └─ 파일 겹침? → 순차 처리
  │
  ├─ 구조 파악/탐색 필요?
  │   ├─ 10파일 이상 읽어야? → context-gatherer 위임
  │   └─ 특정 파일 몇 개만? → main 직접 읽기
  │
  ├─ 기획/디자인/아키텍처 결정?
  │   └─ multi-perspective-review
  │
  └─ 위 어디에도 해당 안 함?
      └─ main 직접 처리
```

## 서브에이전트: 위임 vs 직접 기준

### 서브에이전트에 위임해야 하는 경우

| 조건 | 이유 |
|------|------|
| 전문 역할의 프롬프트가 필요 (리뷰어, 기획자, 디자이너) | 역할별 페르소나 + 체크리스트가 있어야 품질 높은 결과 |
| 대량 코드 수정 (10파일+, 100줄+) | main 컨텍스트를 보호하면서 작업 위임 |
| FE + BE 동시 변경 | 병렬 실행으로 시간 단축 |
| 결과물 형식이 정해진 작업 | 에이전트가 일관된 형식으로 산출물 생성 |

### main이 직접 해야 하는 경우

| 조건 | 이유 |
|------|------|
| 1~2파일, 10줄 이내 수정 | 서브에이전트 호출 오버헤드가 더 큼 |
| 설정 파일 수정 (steering, .env 등) | 간단하고 컨텍스트 전달 불필요 |
| 사용자와 대화하며 방향 결정 중 | 서브에이전트는 대화 못 함 |
| 이전 턴 결과를 바로 이어서 수정 | 컨텍스트가 이미 main에 있음 |

## git worktree: 사용 판단 기준

### worktree 분리 제안할 때

| 조건 | 모두 충족 시 제안 |
|------|-----------------|
| 현재 진행 중인 작업이 있음 | ✅ |
| 새 작업이 현재 작업과 **파일이 겹치지 않음** | ✅ |
| 새 작업이 **30분 이상** 소요 예상 | ✅ |
| 새 작업이 **별도 브랜치**에서 해야 함 | ✅ |

### worktree 사용하지 않을 때

| 조건 | 이유 |
|------|------|
| 5분 이내 끝나는 작업 | stash → 처리 → pop이 더 빠름 |
| 같은 파일 수정 필요 | 충돌 불가피 → 순차 처리 |
| 단일 브랜치에서 해도 되는 작업 | 분리 오버헤드 불필요 |

## 상황 감지 → 제안 패턴

AI가 작업 중 아래 상황을 감지하면 사용자에게 도구 사용을 **제안**한다 (강제 아님):

| 감지 패턴 | 제안 |
|-----------|------|
| "이거 하는 동안 저것도 해줘" (동시 작업 요청) | "현재 작업과 독립적이면 git worktree로 분리할까요?" |
| FE + BE 파일 동시 수정이 예상됨 | "FE/BE 병렬로 서브에이전트에 위임할까요?" |
| 처음 보는 코드베이스 + 넓은 범위 파악 필요 | "context-gatherer로 구조 파악 먼저 할까요?" |
| 기획/디자인 방향에서 2가지 이상 선택지 | "다관점 심의 적용할까요?" |
| 수정 완료 후 변경량이 크다 | "코드 리뷰어에게 검토 맡길까요?" |

> 제안은 한 줄로 짧게. 사용자가 "ㅇㅇ" 하면 실행, "아니" 하면 넘어감.

## 도구 간 상하 관계

```
work-strategy (상위: 무슨 도구를 쓸지 판단)
  ├── subagents-collaboration (서브에이전트 호출 절차/규칙)
  ├── parallel-work (git worktree 절차/규칙)
  ├── context-efficiency (컨텍스트 사용량 관리)
  ├── verification-loop (검증 사이클)
  └── multi-perspective-review (다관점 심의)
```

- `work-strategy`는 "무슨 도구를 꺼낼지" 결정
- 각 하위 자산은 "꺼낸 도구를 어떻게 쓰는지" 상세 절차

## 조합 예시

| 시나리오 | 도구 조합 |
|---------|-----------|
| 새 기능 풀스택 구현 | Phase 0(영향 분석) → product-planner → ui-designer → FE+BE 서브에이전트 병렬 → verification-loop → 코드 리뷰어 |
| 긴급 핫픽스 + 현재 기능 개발 중 | worktree 분리 → 핫픽스 수정 → verification-loop → merge → worktree 제거 → 원래 작업 계속 |
| 대규모 리팩토링 | context-gatherer(구조 파악) → 영향 분석 → 단계별 서브에이전트 위임 → 매 단계 verification-loop |
| 비즈니스 규칙 변경 | multi-perspective-review → Phase 0 → 문서 수정 → 코드 수정 → verification-loop → QA |
| 단순 버그 수정 (1파일) | main 직접 수정 → verification-loop |
| 기획 논의 (구현 없음) | multi-perspective-review → product-planner |

## 안티패턴 (하지 말 것)

| 안티패턴 | 문제 | 대안 |
|---------|------|------|
| 5줄 수정에 서브에이전트 호출 | 오버헤드 > 이점 | main 직접 |
| 5분 작업에 worktree 생성 | 생성/정리 시간이 작업보다 김 | stash 또는 그냥 순차 |
| 매번 context-gatherer 호출 | 컨텍스트 낭비 | 대상 파일 명확하면 직접 읽기 |
| 모든 결정에 multi-perspective | 분석 시간 낭비 | 단순 구현은 바로 진행 |
| 도구 여러 개 동시 활성화 | 복잡도 폭발 | 한 번에 하나씩, 완료 후 다음 |
