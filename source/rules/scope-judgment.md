# 스코프 판단 기준

## 목적

기능 구현 중 "이것도 고쳐야 할 것 같은데..." 발견할 때, 지금 할지 나중으로 미룰지 판단하는 기준.
AI가 스코프를 무한 확장하거나, 반대로 명백한 문제를 무시하는 것을 방지.

---

## 판단 테이블

| 조건 | 행동 |
|------|------|
| 현재 작업과 직접 연결됨 (호출 관계, 같은 기능, 같은 API) | ✅ 같이 수정 |
| 현재 작업과 무관하지만 1줄 수정으로 끝남 | ⚡ 같이 수정 가능 (별도 커밋으로 분리) |
| 현재 작업과 무관 + 수정 범위가 큼 (3줄+) | ❌ TODO 주석 또는 메모만. 지금 안 함 |
| 수정하면 현재 작업의 빌드/테스트에 영향 | ❌ 절대 지금 안 함. 별도 작업으로 분리 |
| 보안 취약점 발견 | ⚠️ 사용자에게 즉시 보고. 심각도에 따라 판단 요청 |

---

## "지금 할 것" 기준

다음 조건을 **모두** 만족할 때만 현재 작업 범위에 포함:
1. 현재 변경하는 파일/기능과 직접 관련 있음
2. 안 하면 현재 작업 결과가 정상 동작하지 않거나 불완전함
3. 추가 작업량이 현재 작업의 20% 이내

---

## "나중에 할 것" 처리 방법

| 방법 | 언제 |
|------|------|
| TODO 주석 (`// TODO: {설명}`) | 코드 내 특정 위치에 맥락 필요 시 |
| 사용자에게 보고 | "이런 문제도 보이는데, 별도로 처리할까요?" |
| 별도 커밋으로 분리 | 1줄 수정이라 지금 해도 되지만 논리적 분리 필요 시 |

---

## 안티패턴

| 패턴 | 문제 |
|------|------|
| "여기 보니까 이것도 고칠게요" (무한 확장) | 스코프 크립. 원래 작업이 끝나지 않음 |
| "요청받은 부분만 했습니다" (명백한 문제 무시) | 바로 옆에 빌드 에러/타입 에러가 있는데 무시 |
| 리팩토링 + 기능 추가 동시 | 뭐가 문제인지 추적 불가, revert 불가능 |

---

## 사용자에게 보고하는 경우

현재 작업 범위 밖에서 발견한 것 중 보고할 가치가 있는 것:
- 보안 취약점
- 데이터 유실 가능성
- 빌드가 깨지는 문제
- 반복되는 패턴 (playbook 축적 대상?)

보고 형식:
> "참고: {파일명}에서 {문제}를 발견했습니다. 현재 작업과 무관하므로 별도로 처리하면 좋겠습니다."
