# 코드 읽는 순서

## 목적

프로젝트를 처음 보거나, 특정 기능을 이해해야 할 때 무엇부터 읽으면 빠르게 파악할 수 있는지.
AI가 무작위로 파일을 읽는 대신, 효율적인 순서를 따르도록.

---

## 프로젝트 전체 파악 (처음 보는 코드베이스)

```
1. product.md / README → "이 프로젝트가 뭔지"
2. structure.md / 패키지 구조 → "어디에 뭐가 있는지"
3. 진입점 (Application.java, index.ts, main.py) → "어떻게 시작하는지"
4. 라우팅/엔드포인트 목록 → "뭘 할 수 있는지"
5. 핵심 도메인 모델 (Entity, Type) → "무슨 데이터를 다루는지"
```

---

## 특정 기능 이해 (백엔드)

```
1. API 엔드포인트 (Controller) → "입구가 어디인지"
2. Service → "비즈니스 로직이 뭔지"
3. Repository / DB 접근 → "데이터를 어떻게 다루는지"
4. DTO → "입출력 형태가 뭔지"
5. Infra/Client (외부 연동) → "외부 시스템과 어떻게 통신하는지"
```

---

## 특정 화면 이해 (프론트엔드)

```
1. cases.md → "이 화면이 뭘 하는지, 어떤 상태가 있는지"
2. HTML 디자인 명세 → "어떻게 보여야 하는지"
3. 페이지 컴포넌트 → "실제 구현"
4. hooks / stores → "상태 관리"
5. services (API 호출) → "서버와 어떻게 통신하는지"
```

---

## 버그 추적

```
1. 에러 메시지/로그 → "무슨 에러인지"
2. 스택 트레이스 → "어디서 발생했는지"
3. 해당 파일의 에러 발생 지점 → "왜 발생했는지"
4. 호출부 (이 함수를 부르는 곳) → "어떤 입력이 문제인지"
5. 관련 테스트 → "이 케이스가 커버되고 있었는지"
```

---

## 영향 분석 (변경 전)

```
1. 변경 대상 파일 → "뭘 바꿀 건지"
2. 이 파일을 import/호출하는 곳 → "누가 쓰고 있는지"
3. 관련 테스트 → "뭘 검증하고 있는지"
4. 관련 문서 (cases.md, business-logic) → "기획과 일치하는지"
```

---

## 읽지 않아도 되는 것

| 상황 | 안 읽어도 됨 |
|------|-------------|
| 특정 기능 수정 시 | 무관한 다른 기능의 코드 |
| API 수정 시 | 프론트엔드 전체 (영향받는 호출부만) |
| 설정 변경 시 | 비즈니스 로직 |
| 단순 스타일 변경 시 | 서버 코드 |

---

## context-efficiency와의 관계

- 이 순서를 따르되, `context-efficiency`의 "같은 파일 2회 읽지 않기", "탐색 10개+ 시 context-gatherer 위임" 규칙도 함께 적용
- 순서대로 읽되, 필요 없다고 판단되면 단계를 건너뛰어도 됨
