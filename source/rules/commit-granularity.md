# 커밋 분리 기준

## 핵심 원칙

> "이 커밋을 revert했을 때 의미 있는 단위인가?"

각 커밋은:
- 단독으로 빌드가 통과해야 한다
- 하나의 논리적 변경을 담아야 한다
- revert 시 부작용 없이 되돌릴 수 있어야 한다

---

## 같은 커밋에 넣어도 되는 것

| 조합 | 이유 |
|------|------|
| 기능 추가 + 해당 기능 테스트 | 기능과 테스트는 한 세트 |
| 버그 수정 + 회귀 방지 테스트 | 수정과 검증은 한 세트 |
| 같은 기능의 FE + BE | 하나의 기능 단위 (프로젝트 규모에 따라 분리도 OK) |
| DTO 변경 + 사용하는 코드 수정 | 인터페이스 변경은 한번에 |
| import 정리 + 해당 파일 수정 | 같은 파일 정리 |

## 반드시 분리해야 하는 것

| 조합 | 이유 |
|------|------|
| 기능 추가 + 무관한 리팩토링 | feat + refactor 혼합 시 revert 불가능 |
| 버그 수정 + 새 기능 | 버그 수정만 cherry-pick 할 수 없게 됨 |
| 대규모 포맷팅 + 로직 변경 | diff에서 실제 변경을 찾을 수 없음 |
| DB 마이그레이션 + 코드 변경 | 롤백 순서가 달라야 할 수 있음 |
| 의존성 업데이트 + 코드 변경 | 의존성 문제 시 분리 롤백 필요 |

---

## 커밋 크기 가이드

| 크기 | 적절한가? |
|------|-----------|
| 1~5 파일, 하나의 변경 목적 | ✅ 이상적 |
| 5~15 파일, 하나의 기능 | ✅ 허용 (규모 큰 기능) |
| 15파일+, 여러 목적 혼합 | ❌ 분리 필요 |
| 1줄 수정 | ✅ OK (typo, 설정 변경 등) |

---

## 실전 시나리오

### "리팩토링 하다가 버그 발견"
1. 리팩토링 중단 → 현재까지 작업 커밋 (refactor)
2. 버그 수정 → 별도 커밋 (fix)
3. 리팩토링 이어서 → 별도 커밋 (refactor)

### "기능 만드는데 기존 코드 정리도 필요"
1. 기존 코드 정리 먼저 → 커밋 (refactor)
2. 새 기능 추가 → 커밋 (feat)

### "설정 변경 + 코드 변경"
- 설정만으로 의미 있는 단위면 → 분리 (chore + feat)
- 설정과 코드가 한 세트면 → 같이 (feat)
