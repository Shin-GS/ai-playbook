# 명령어 단축키

## 목적

자주 사용하는 워크플로우를 명령어 한 줄로 빠르게 시작한다.
명령어는 **단축키일 뿐** — 명령어 없이 자연어로 해도 동일한 결과가 나와야 한다.

> 이 명령어들은 기존 workflow/rule을 "명시적으로 트리거"하는 인터페이스. 새 로직을 추가하는 것이 아님.

---

## 명령어 목록

| 명령 | 동작 | 참조 자산 |
|------|------|----------|
| `/기획 {설명}` | 기획 워크플로우 | subagents-collaboration Phase 1 + 1.5 |
| `/디자인 {설명}` | 디자인 워크플로우 | subagents-collaboration Phase 2 + 2.5 |
| `/리뷰` | 현재 변경사항 리뷰 | subagents-collaboration Phase 4 |
| `/배포전` | 배포 전 종합 체크 | verification-loop + 문서 동기화 + QA |
| `/영향분석 {설명}` | 영향 범위 분석 | subagents-collaboration Phase 0 |
| `/만화 {설명}` | 만화/일러스트 생성 | image-generation |
| `/사진편집 {설명}` | 이미지 편집 | image-editing |

## /기획 {설명}

사용자가 기능을 자연어로 설명하면 기획 워크플로우를 실행한다.

### 동작 절차

1. **판단**: 기능 설명 분석
   - 기존 화면 수정인가? → 해당 cases.md 찾기
   - 새 화면/기능인가? → 화면명 결정 + cases.md 신규 생성
   - 여러 화면에 걸치는가? → 영향받는 화면 목록 도출
   - 판단 불가 시 → 사용자에게 되묻기

2. **실행**: `product-planner` 에이전트 호출
   - 참조: business-logic.md, glossary.md, 기존 cases.md
   - 산출물: cases.md (신규 또는 수정)

3. **리뷰** (변경 규모 클 때): `planning-reviewer` 호출
   - 빠진 케이스, 모순, 엣지케이스 검증
   - 피드백 있으면 → product-planner에게 수정 지시
   - 1회 리뷰 원칙

4. **완료 후 자동 제안**: "디자인도 이어서 할까요?"
   - 승인 시 → /디자인 자동 실행
   - 거절 시 → 여기서 끝

### 예시

```
/기획 사용자가 결제 후 영수증을 볼 수 있게 해줘
/기획 장바구니에 쿠폰 적용 기능
/기획 기존 로그인 화면에 소셜 로그인 추가
```

## /디자인 {설명}

디자인 워크플로우를 실행한다. cases.md 기반으로 HTML 명세를 생성/수정.

### 동작 절차

1. **판단**: 기능 설명에서 대상 화면 파악
   - 해당 cases.md 존재 확인 (없으면 "기획이 먼저 필요합니다" 안내)
   - 해당 HTML 존재 여부 → 신규 생성 or 수정

2. **전제 조건 확인**:
   - 디자인 시스템(tokens.css, components.css) 있는지
   - 없으면 → design-system workflow의 Step 0 먼저 실행

3. **실행**: `ui-designer` 에이전트 호출
   - 참조: cases.md, tokens.css, 기존 HTML 2~3개 (톤앤매너)
   - 산출물: {화면}.html

4. **리뷰** (신규 화면일 때): `design-reviewer` 호출
   - 접근성, 토큰 일관성, 케이스 커버리지 검증
   - 1회 리뷰 원칙

5. **완료 후 자동 제안**: "FE 구현도 할까요?"

### 예시

```
/디자인 영수증 화면
/디자인 장바구니 수량 변경 부분 수정
/디자인 로그인에 소셜 버튼 추가
```

## /리뷰

현재 변경사항에 맞는 리뷰어를 자동 호출한다.

### 동작 절차

1. **변경 감지**: git diff (staged 우선, 없으면 unstaged)
   - 변경 없으면 → "변경사항이 없습니다. 뭘 리뷰할까요?" 되묻기

2. **리뷰어 자동 선택**:
   | 변경 파일 | 리뷰어 |
   |-----------|--------|
   | *.java, *.kts | backend-code-reviewer |
   | *.ts, *.tsx | frontend-code-reviewer |
   | *.js (Node.js 프로젝트) | code-reviewer-nodejs |
   | cases.md, business-logic.md | planning-reviewer |
   | docs/design/*.html | design-reviewer |
   | 여러 영역 혼합 | 각각 병렬 호출 |

3. **확인** (여러 영역일 때): "FE + BE 변경입니다. 둘 다 리뷰할까요?"

4. **결과 종합 보고**

### 예시

```
/리뷰
/리뷰  (← 인자 없이 사용)
```

## /배포전

배포 전 종합 체크를 실행한다.

### 동작 절차

1. **verification-loop**: 빌드 → 타입 체크 → 린트 → 테스트
2. **문서 동기화 체크**: 코드 변경에 맞게 문서 갱신됐는지 (project-docs-structure 매핑 테이블 기준)
3. **QA 체크리스트 확인**: 변경된 기능의 테스트 데이터가 최신인지
4. **git 상태 확인**: 커밋 안 한 변경, untracked 파일

### 결과 보고 형식

```
## 배포 전 체크

- 빌드: ✅ 통과
- 타입 체크: ✅ 통과
- 린트: ✅ 통과 (경고 2건)
- 테스트: ✅ 12/12 통과
- 문서 동기화: ✅ 최신
- QA 체크리스트: ⚠️ payment.js 미갱신 (결제 로직 변경됨)
- git 상태: ✅ clean

→ QA 체크리스트 갱신 후 배포 권장
```

### 예시

```
/배포전
```

## /영향분석 {설명}

특정 변경을 하기 전에 영향 범위를 분석한다. subagents-collaboration의 Phase 0을 단독 실행.

### 동작 절차

1. **변경 설명 분석**: 뭘 바꾸려는지 파악
2. **영향 분석 실행**: Phase 0 절차 그대로
   - 코드, 문서, 테스트, API 영향 확인
   - 규모별 분석 깊이 적용
3. **보고**: 영향 분석 보고서 형식으로 결과 제공
4. **"진행할까요?"** — 사용자 승인 대기

### 예시

```
/영향분석 결제를 카카오페이에서 토스로 변경
/영향분석 사용자 테이블에 phone 컬럼 추가
/영향분석 인증 방식을 JWT에서 세션으로 변경
```

## /만화 {설명}

만화/일러스트 생성 파이프라인을 실행한다.

> 이 명령 실행 시 `source/skills/creative/image-generation.md` 를 반드시 읽고 파이프라인을 따를 것.

### 동작 절차

1. **확인**: 작업 전 확인 사항 (image-generation skill 기준)
   - 타겟 플랫폼 → 사이즈 자동 결정
   - 스타일 → 기존 style-guide.md 있으면 참조, 없으면 정의
   - 캐릭터 → 기존 시트 있으면 참조, 없으면 생성
   - 용도 → SNS? 앱? 마케팅?

2. **문서 생성/확인**: world.md, style-guide.md, characters/ 참조 또는 생성

3. **지시서 작성**: projects/{프로젝트}/brief.md 또는 ep{N}.md 생성

4. **아스키 프로토타입**: 콘솔 스타일 아스키 아트로 레이아웃 시각화
   - 사용자 확인 → 수정 → 반복

5. **코드 생성·실행**: 프로토타입 + 지시서 기반 이미지 코드 (SVG, Pillow, Sharp 등)

6. **검증**: 사이즈, 텍스트 잘림, 스타일 일관성 자동 체크

7. **출력**: assets/output/ 에 저장

### 예시

```
/만화 인스타그램용 4컷, 재택근무하는 고양이
/만화 유튜브 썸네일, "오늘의 코딩 실수 TOP3"
/만화 캐릭터 토리로 새 에피소드, 비 오는 날 카페
```

## /사진편집 {설명}

이미지 편집 파이프라인을 실행한다.

> 이 명령 실행 시 `source/skills/creative/image-editing.md` 를 반드시 읽고 파이프라인을 따를 것.

### 동작 절차

1. **확인**: 작업 전 확인 사항 (image-editing skill 기준)
   - 원본 이미지 위치
   - 수정본 저장 위치
   - 타겟 플랫폼/용도 → 사이즈·포맷 자동 결정
   - 편집 종류 (리사이즈, 필터, 합성, 워터마크 등)
   - 배치 여부 (단일 vs 다수)

2. **편집 지시서 생성** (복잡한 작업 시)

3. **1개 파일 테스트** (배치 시): 결과 확인 후 전체 적용

4. **코드 생성·실행**: Sharp, Pillow, ImageMagick 등

5. **검증**: 원본 무변경, 사이즈, 포맷, 용량 체크

6. **결과 보고**: 성공/실패 목록

### 예시

```
/사진편집 assets/original/ 이미지들을 인스타 피드 사이즈로 리사이즈
/사진편집 hero.png에 로고 워터마크 추가
/사진편집 product/ 폴더 전체 배경 제거
```

## 규칙

### 명령어 없이도 동일 동작

아래 두 표현은 동일한 결과를 내야 한다:
- `/기획 영수증 기능 추가해줘`
- `영수증 기능을 기획해줘`

명령어는 "이건 기획 워크플로우로 가라"는 명시적 신호. 없어도 AI가 맥락에서 판단 가능하면 동일하게 실행.

### 복합 명령어 금지

`/기획+디자인` 같은 복합 명령어를 만들지 않는다. 연쇄 실행은 "완료 후 자동 제안"으로 자연스럽게.

### 프로젝트별 확장

프로젝트 특화 명령어가 필요하면 프로젝트의 .kiro/steering/에 별도로 추가. playbook의 이 파일을 수정하지 않는다.
