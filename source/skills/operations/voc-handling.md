# 사용자 문의 대응 (VOC)

## 목적

1인 빌더가 개발하면서 사용자 문의도 효율적으로 처리하기 위한 가이드.
"모든 문의에 즉시 응답"이 아니라 "분류 → 우선순위 → 적절한 시간 내 대응"이 목표.

---

## 문의 분류 및 우선순위

| 유형 | 예시 | 우선순위 | 대응 시간 |
|------|------|---------|-----------|
| 🔴 긴급 버그 | "결제했는데 반영 안 됨", "데이터 사라짐" | P0 | 즉시 (1시간 이내) |
| 🟡 일반 버그 | "버튼 안 눌려요", "페이지 에러" | P1 | 당일 |
| 🔵 사용법 문의 | "이거 어떻게 해요?" | P2 | 24시간 이내 |
| 🟢 기능 요청 | "이런 기능 있으면 좋겠어요" | P3 | 주 단위 검토 |
| ⚪ 불만/피드백 | "느려요", "불편해요" | P2 | 24시간 이내 |

---

## 대응 원칙 (1인 운영)

### 시간 관리
- **개발 시간 보호**: CS 전용 시간대를 정함 (예: 오전 30분, 오후 30분)
- **P0 외에는 즉시 대응하지 않음**: 개발 흐름 유지가 더 중요
- **"확인 중" 즉시 응답**: 무응답이 최악. 확인 중이라도 먼저 알림

### 응답 톤
- 공감 먼저: "불편을 드려 죄송합니다"
- 구체적으로: "확인 결과 ~때문에 발생했습니다"
- 해결책 제시: "~하시면 됩니다" 또는 "수정 배포 예정입니다"
- 짧게: 장문 금지, 3~5줄 이내

### 하지 말 것
- 기술적 상세를 사용자에게 설명 ("DB 커넥션 풀이..." ❌)
- 변명 ("혼자 개발해서..." ❌)
- 과잉 약속 ("내일까지 만들어드릴게요" — 못 지키면 신뢰 하락)

## 버그 리포트 → 이슈 전환 절차

### 재현 가능 여부 확인

사용자에게 물어볼 것:
- 어떤 기기/브라우저에서?
- 어떤 순서로 조작했는지?
- 스크린샷/에러 메시지 있으면?
- 언제부터 발생? (매번? 가끔?)

### 전환 절차
1. 재현 확인 → 이슈 생성 (우선순위 태깅)
2. 재현 안 됨 → 추가 정보 요청 or "모니터링 중" 안내
3. 수정 완료 → 사용자에게 "수정했습니다" 응답
4. QA 체크리스트에 regression 마킹

## VOC → 기획 반영 기준

| 조건 | 행동 |
|------|------|
| 같은 문의 1회 | 메모만 |
| 같은 문의 3회+ | UX 개선 또는 가이드/FAQ 추가 검토 |
| 돈/보안 관련 | 즉시 대응 (횟수 무관) |
| "이런 기능 있으면" 1회 | 메모만 (바로 만들지 않음) |
| "이런 기능 있으면" 3회+ | 기획 검토 대상으로 등록 |
| 사용법 문의 반복 | UI 자체가 직관적이지 않은 것 → 개선 대상 |

## 반복 문의 축적 → FAQ

### FAQ 생성 기준
- 같은 질문 2회 이상 → FAQ 항목으로 작성
- FAQ 항목은 제품 내 노출 (Help/FAQ 페이지) + 응답 시 링크 전달

### FAQ 형식
```markdown
**Q: {사용자가 물어보는 형태 그대로}**
A: {짧고 구체적인 답변}
```

---

## 자동화 고려 (여유 있을 때)

### 지금 당장 할 수 있는 것
- 상태 페이지 (서비스 정상/장애 표시) — 정적 HTML이면 충분
- 자동 응답 봇 (반복 질문에 FAQ 링크 전달)
- 에러 알림 → 자동 이슈 생성 (GitHub Actions or 간단 스크립트)

### 나중에 고려할 것
- 챗봇 (AI 기반 FAQ 응답)
- 피드백 수집 폼 (구조화된 버그 리포트)
- 사용자 행동 로그 기반 문제 자동 감지

---

## 커뮤니케이션 채널 관리

### 채널 선택 기준 (1인 빌더)
- **1개만 유지**: 여러 채널 모니터링은 비현실적
- 추천: Telegram 채널/그룹 (알림 즉시, 모바일 대응 가능) 또는 Discord
- 이메일은 보조 (긴급 아닌 것만)

### 채널 운영 규칙
- 공지 전용 채널 + 문의/대화 채널 분리
- 문의 채널에 "응답 시간: 24시간 이내" 명시 (기대치 관리)
- 장애 시 공지 채널에 상태 업데이트
