# 장애 대응 절차

## 목적

서비스 장애 발생 시 패닉 상태에서도 따라갈 수 있는 체크리스트.
1인 빌더 환경의 특수 상황(수면 중, 외출 중, 다른 작업 중)도 고려한다.

---

## 심각도 분류 (먼저 판단)

| 레벨 | 상태 | 예시 | 대응 시간 |
|------|------|------|-----------|
| P0 | 서비스 완전 다운 | 서버 응답 없음, DB 연결 불가 | 즉시 |
| P1 | 핵심 기능 장애 | 결제 실패, 로그인 불가, 데이터 유실 | 1시간 이내 |
| P2 | 부분 기능 장애 | 특정 페이지 에러, 알림 미발송 | 당일 |
| P3 | 경미한 이슈 | UI 깨짐, 로그 경고 | 다음 작업 시 |

---

## 1인 특수 상황 판단

### 다른 작업 중 장애 알림 수신 시
- P0/P1: 현재 작업 즉시 중단. 변경 사항 stash/커밋 후 장애 대응.
- P2: 현재 작업 마무리 후 대응 (30분 이내).
- P3: 메모만 하고 현재 작업 계속.

### 수면 중 / 외출 중
- P0: 자동 복구 불가면 → 알림에 "즉시 확인 필요" 포함해야 함. 모바일에서 할 수 있는 것(서버 재시작, 롤백)만 시도.
- P1~P3: 아침/귀가 후 대응. 사용자 영향 있으면 상태 공지만 우선.

### 모바일에서 할 수 있는 것
- SSH 앱으로 서버 상태 확인
- Docker 컨테이너 재시작
- 이전 배포 버전으로 롤백 (사전 스크립트 준비 필요)
- 사용자에게 상태 공지

### 반드시 PC가 필요한 것
- 코드 수정 (핫픽스)
- DB 복구
- 복잡한 원인 분석

---

## 대응 절차 (P0/P1)

### Step 1: 상태 확인 (5분 이내)

```bash
# 서비스 살아있나?
curl -f https://도메인/health

# 프로세스 상태
docker ps              # 또는 pm2 status, systemctl status

# 최근 로그
docker logs --tail 50 {container}
```

- **done-when**: 서비스 상태(다운/부분장애/성능저하) 파악됨
- **fail-action**: SSH 접속 자체가 안 되면 → 호스팅 업체 상태 페이지 확인 / 다른 서버에서 ping

### Step 2: 빠른 복구 시도 (15분 이내)

| 증상 | 즉시 조치 |
|------|-----------|
| 서비스 다운 | 컨테이너/프로세스 재시작 |
| 최근 배포 후 장애 | 이전 버전으로 롤백 |
| DB 연결 실패 | DB 서비스 상태 확인 + 재시작 |
| 디스크 풀 | 로그/임시 파일 정리 |
| 메모리 부족 | 프로세스 재시작 + OOM 원인 파악 |

- **done-when**: 서비스 정상 복구됨
- **fail-action**: 재시작/롤백으로 안 되면 → Step 3 원인 분석

### Step 3: 원인 분석

1. **최근 변경 확인**: `git log --oneline -10`, 배포 기록
2. **에러 로그 분석**: 시간순 역추적, 첫 에러 시점 찾기
3. **리소스 확인**: CPU, 메모리, 디스크, 네트워크, DB 커넥션
4. **외부 서비스**: 연동 API 상태, DNS, CDN

### Step 4: 핫픽스 또는 워크어라운드

- 핫픽스: 최소 변경으로 수정 → 로컬 테스트 → 배포
- 워크어라운드: 임시 우회 (기능 비활성화, 캐시 비우기 등)
- 어느 쪽이든: 수정 후 모니터링 (재발 여부 확인)

### Step 5: 사후 분석 (복구 후 24시간 이내)

```markdown
## 장애 보고서

### 타임라인
- HH:MM 장애 발생 (증상)
- HH:MM 알림 수신
- HH:MM 원인 파악
- HH:MM 복구 완료

### 근본 원인
- (왜 발생했는지)

### 영향
- 영향 시간: N분
- 영향 사용자: 추정 N명
- 데이터 유실: 있음/없음

### 재발 방지
- [ ] (모니터링 추가)
- [ ] (테스트 추가)
- [ ] (규칙 추가 — playbook 축적 대상?)
```

## 자동 복구 vs 수동 복구 구분

사전에 세팅해두면 P0에서 사람 없이도 복구 가능한 것:

| 자동 복구 가능 | 방법 |
|--------------|------|
| 프로세스 크래시 | Docker restart policy: always, pm2 자동 재시작 |
| 일시적 메모리 부족 | 컨테이너 메모리 제한 + 자동 재시작 |
| 헬스체크 실패 | 로드밸런서/Docker healthcheck + 자동 재시작 |

| 수동 대응 필수 | 이유 |
|--------------|------|
| 코드 버그 | 수정 필요 |
| DB 데이터 손상 | 복구 판단 필요 |
| 외부 API 장기 장애 | 우회 또는 대기 판단 |
| 디스크 풀 | 뭘 지울지 판단 |

## 사용자 커뮤니케이션

### P0/P1 시

```
[장애 인지 시]
"현재 서비스에 일시적인 문제가 발생했습니다. 확인 중이며, 복구되는 대로 안내드리겠습니다."

[복구 완료 시]
"서비스가 정상 복구되었습니다. 이용에 불편을 드려 죄송합니다."
```

- 기술적 상세는 넣지 않음 (사용자한테 "DB 커넥션 풀 고갈" 말해봐야 의미 없음)
- 복구 예상 시간을 알면 포함, 모르면 "확인 중"
- 데이터 유실이 있으면 솔직하게 안내 + 보상 방안
