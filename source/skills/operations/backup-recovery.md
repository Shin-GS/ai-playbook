# 백업/복구 전략

## 목적

"데이터 잃기 전에" 세팅하는 가이드. 백업은 해두고, 복구는 테스트해봐야 의미 있다.
1인 빌더 환경에서 비용과 복잡도를 최소화하면서 핵심 데이터를 보호한다.

---

## 백업 대상 분류

| 대상 | 백업 필수? | 이유 |
|------|-----------|------|
| 코드 | ✅ (Git으로 이미 됨) | 별도 조치 불필요 |
| DB 데이터 | ✅ 필수 | 사용자 데이터, 트랜잭션 — 유실 시 치명적 |
| 업로드 파일 (이미지 등) | ✅ 필수 | 사용자 생성 콘텐츠 — 재생성 불가 |
| 환경 설정 (.env, 인프라) | ✅ 필수 | 없으면 서비스 재구성 불가 |
| 캐시 (Redis 등) | ❌ | 재생성 가능 |
| 세션 데이터 | ❌ | 재로그인하면 됨 |
| 빌드 산출물 | ❌ | 코드에서 재빌드 가능 |
| 로그 | ⚡ 선택 | 장애 분석용으로 보관하면 좋음 |

---

## 백업 주기 기준

| 데이터 성격 | 권장 주기 | 보관 기간 | 이유 |
|------------|----------|-----------|------|
| 돈/결제 관련 | 실시간 복제 또는 1시간 | 90일+ | 법적 요구, 분쟁 대응 |
| 사용자 데이터 | 일 1회 | 30일 | 하루치 유실은 감수 가능 |
| 업로드 파일 | 업로드 즉시 (별도 스토리지) | 무기한 | 원본 유실 시 복구 불가 |
| 환경 설정 | 변경 시 즉시 | 최신 1개 + 이전 1개 | 변경 빈도 낮음 |
| 로그 | 일 1회 압축 | 30~60일 | 디스크 절약 |

## 비용 현실 (1인 빌더)

### 최소 비용 구성

| 방법 | 비용 | 용도 |
|------|------|------|
| DB 덤프 → 같은 서버 다른 디렉토리 | 무료 | 최소한의 로컬 백업 (서버 소실 시 무용) |
| DB 덤프 → S3/B2 (클라우드 오브젝트 스토리지) | ~$1/월 (100GB 미만) | 서버 소실에도 복구 가능 |
| 관리형 DB 자동 백업 (RDS 등) | DB 요금에 포함 | 가장 편함 (이미 쓰고 있으면) |
| 서버 스냅샷 | $1~5/월 | 서버 전체 복제 (가장 간단하지만 큼) |

### "이것만은 반드시" vs "여유 되면"

**반드시:**
- DB 일일 덤프 → 서버 외부에 최소 1곳 저장
- .env 파일 → 암호화해서 별도 보관 (Git 금지)

**여유 되면:**
- 포인트-인-타임 복구 (RDS PITR 등)
- 업로드 파일 CDN + 원본 별도 보관
- 백업 암호화 (전송 + 저장)
- 다른 리전에 백업 복제

## 복구 테스트 (핵심)

> 백업만 하고 복구 안 해보면 의미 없다.

### 월 1회 복구 테스트 절차

1. 최신 백업 파일 다운로드
2. 로컬 또는 별도 환경에서 DB 복원
3. 복원된 DB에서 기본 쿼리 실행 (데이터 존재 확인)
4. 최근 데이터가 포함되어 있는지 확인 (타임스탬프 체크)
5. 복구 소요 시간 측정 및 기록

### 측정할 것

| 항목 | 의미 |
|------|------|
| RPO (Recovery Point Objective) | 최대 몇 시간/일치 데이터를 잃을 수 있는가 |
| RTO (Recovery Time Objective) | 복구에 최대 얼마나 걸리는가 |

1인 빌더 현실적 목표:
- RPO: 24시간 이내 (일일 백업 기준)
- RTO: 2시간 이내 (새 서버 + DB 복원)

## 복구 절차 (시나리오별)

### 시나리오 1: DB 데이터 손상/삭제

1. 서비스 중단 (추가 손상 방지)
2. 최신 백업 위치 확인
3. 현재 DB 스냅샷 보존 (분석용)
4. 백업으로 복원
5. 복원 후 정합성 확인 (레코드 수, 최신 데이터 타임스탬프)
6. 서비스 재시작 + 헬스체크
7. 유실 데이터 범위 파악 → 사용자 안내 (필요 시)

### 시나리오 2: 서버 자체 소실

1. 새 서버 프로비저닝
2. 기본 환경 세팅 (Docker, Nginx 등)
3. Git clone (코드)
4. .env 복원 (별도 보관 위치에서)
5. DB 복원 (클라우드 백업에서)
6. 업로드 파일 복원 (S3/CDN에서)
7. 서비스 시작 + 헬스체크 + DNS 전환 (IP 변경 시)

### 시나리오 3: 실수로 데이터 삭제 (특정 레코드)

1. 삭제 시점 파악
2. 포인트-인-타임 복구 가능하면 → 해당 시점으로
3. 불가능하면 → 일일 백업에서 해당 테이블만 추출
4. 추출한 데이터를 현재 DB에 선택적 복원

## 백업 보안

- 백업 파일도 공격 대상 — 접근 권한 제한 필수
- 클라우드 저장 시 암호화 (S3 SSE, 또는 업로드 전 gpg)
- 백업 스토리지 접근 키는 서비스 키와 분리
- 백업 보관 위치를 서비스 서버와 같은 계정/같은 서버에 두지 않음
