# 만화 연재 시스템

## 핵심 전제

> 이 skill은 만화 시리즈의 "기획/관리 체계"를 정의한다.
> 실제 이미지 생성은 image-generation skill의 파이프라인을 따른다.
> 처음에 아무것도 없는 상태에서 대화를 통해 전체 구조를 만들 수 있어야 한다.

---

## 디렉토리 구조

```
input/{시리즈명}/
├── series.md              ← 시리즈 메타 (플랫폼, 그림체, 고정 설정)
├── story-bible.md         ← 타임라인, 복선, 확정 설정, 금기
├── characters/
│   ├── _index.md          ← 캐릭터 목록 + 관계도
│   └── {이름}.md          ← 개별 캐릭터 상세
├── assets/
│   ├── style-ref.png      ← 그림체 참고 이미지
│   └── characters/        ← 캐릭터별 참고 이미지
└── episodes/
    ├── ep01/
    │   ├── ep01.md        ← 에피소드 지시서
    │   └── prompts.md     ← 실제 API에 보낸 프롬프트 기록
    └── ep02/
        └── ...

output/{시리즈명}/
├── ep01/
│   ├── page-01.png        ← 최종 출력
│   ├── panel-01-v1.png    ← 패널별 생성 이력 (리트라이 시)
│   └── panel-01-v2.png
└── ep02/
    └── ...
```

## Phase 1. 시리즈 셋업 (처음 시작할 때)

아무것도 없는 상태에서 대화를 통해 구조를 만든다.

### 사전 조건: 이미지 생성 도구 확인

만화 작업을 시작하기 전, 이미지 생성 API가 세팅되어 있는지 확인한다.
세팅되어 있지 않으면 사용자에게 아래 내용을 안내하고 세팅을 도와준다:

```
이미지 생성 API가 세팅되어 있지 않아요.
만화의 실제 그림은 외부 이미지 생성 모델이 담당합니다.

아래 중 하나를 세팅해주세요:

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
방법 1: MCP 서버 연결 (권장 — Kiro에서 도구로 바로 사용)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

.kiro/settings/mcp.json (또는 ~/.kiro/settings/mcp.json)에 추가:

{
  "mcpServers": {
    "image-gen": {
      "command": "npx",
      "args": ["-y", "openrouter-imgen-mcp"],
      "env": {
        "OPENROUTER_API_KEY": "${env:OPENROUTER_API_KEY}"
      }
    }
  }
}

그리고 시스템 환경변수에 OPENROUTER_API_KEY를 설정하세요.
(.env 파일에 직접 키를 적지 마세요 — 유출 위험)

OpenRouter 가입: https://openrouter.ai
API 키 발급: https://openrouter.ai/keys

사용 가능한 모델: FLUX.2 Klein($0.014/장), FLUX.2 Pro($0.03/장),
                  Nano Banana($0.039/장), GPT Image 1 Mini 등

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
방법 2: 코드에서 직접 API 호출
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

MCP 없이도 AI가 스크립트를 작성해서 API를 직접 호출할 수 있습니다.

1. 시스템 환경변수에 API 키 설정:
   - Windows: setx OPENROUTER_API_KEY "sk-or-..."
   - Mac/Linux: export OPENROUTER_API_KEY="sk-or-..."

2. AI가 해당 환경변수를 참조하는 스크립트를 작성·실행합니다.
   (키는 process.env에서 읽으므로 코드에 하드코딩되지 않음)

⚠️ 보안 주의:
- API 키를 .env 파일이나 코드에 직접 적지 마세요
- 시스템 환경변수 또는 시크릿 매니저 사용
- .gitignore에 .env가 포함되어 있는지 확인

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
선택지 요약
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

| 방법 | 장점 | 단점 |
|------|------|------|
| MCP (OpenRouter) | Kiro 도구로 바로 사용, 모델 선택 자유 | MCP 서버 설정 필요 |
| MCP (fal.ai) | FLUX + LoRA 특화, 최저가 | fal.ai 가입 필요 |
| 코드 직접 호출 | 설정 최소, 어디서든 동작 | 매번 스크립트 실행 |

세팅 완료되면 series.md의 '이미지 생성 설정'에 사용할 모델을 기록합니다.
```

> 이미지 생성 도구 없이도 Phase 1~2 (셋업 + 스토리/지시서)는 진행 가능하다.
> Phase 3 (이미지 생성) 시점에 도구가 없으면 다시 안내한다.
> API 키는 반드시 환경변수로 관리하며, 코드나 파일에 직접 기록하지 않는다.

### AI가 물어볼 것

1. **시리즈 이름** (폴더명으로 쓸 영문, 예: wfh-cat)
2. **타겟 플랫폼** (인스타, 웹툰, 유튜브 등)
3. **기본 컷 구성** (4컷, 세로 스크롤, 자유 등)
4. **그림체 참고 이미지** (있으면 첨부 요청)
5. **톤/분위기** (코미디, 감성, 액션 등)
6. **주인공 정보** (이름, 종류, 성격, 참고 이미지)
7. **작업 모드** (careful: 매 단계 확인 / fast: 스토리만 정하고 자동)

### 생성되는 파일

대화 완료 후 AI가 아래 파일들을 자동 생성:
- `input/{시리즈}/series.md`
- `input/{시리즈}/story-bible.md` (빈 템플릿)
- `input/{시리즈}/characters/_index.md`
- `input/{시리즈}/characters/{주인공}.md`
- 첨부된 이미지를 `assets/`에 저장

## series.md 구조

```markdown
# {시리즈명}

## 기본 설정
- 타겟 플랫폼: (instagram / webtoon / youtube 등)
- 사이즈: (1080x1080 등 — 플랫폼에서 자동 결정)
- 컷 구성: (4컷 2x2 / 세로 스크롤 / 자유)
- 톤: (일상 코미디 / 감성 / 액션 / ...)
- 그림체 참고: ./assets/style-ref.png
- 연재 주기: (주 2회 / 미정)

## 이미지 생성 설정
- 모델: (FLUX Klein + LoRA / GPT Image 1 Mini / ...)
- LoRA: (사용할 LoRA 이름, 있으면)
- strength: (참고 이미지 영향력 0.0~1.0)

## 고정 요소
- 말풍선 스타일: (형태, 색상, 테두리)
- 폰트: (대사용, 나레이션용, 효과음용)
- 배경 톤: (색감 키워드)
- 효과음 스타일: (크기, 기울임, 배경 등)

## 작업 모드
- mode: careful  (careful: 매 단계 확인 / fast: 스토리 확정 후 자동)

## 예외
- 사용자가 대화에서 명시적으로 다른 설정을 요청하면 그것을 우선함
```

## story-bible.md 구조

만화가가 "설정 노트"로 관리하는 것들. 연재가 길어질수록 필수.

```markdown
# Story Bible

## 타임라인
| EP | 시간 | 주요 사건 | 등장인물 |
|----|------|-----------|---------|
| 01 | 월요일 아침 | 모치 소개, 첫 재택근무 | 모치 |
| 02 | 화요일 | 옆집 고양이 마루 등장 | 모치, 마루 |

## 복선/떡밥
| 떡밥 | 심은 EP | 회수 EP | 상태 |
|------|--------|---------|------|
| 모치 책상 뒤 택배 상자 쌓임 | 01 | 05 | 미회수 |

## 확정 설정 (변경 금지)
- (에피소드 진행하며 축적)
- 예: "모치는 왼손잡이", "배경은 항상 같은 원룸"

## 금기
- (이 시리즈에서 하면 안 되는 것)
- 예: "인간은 등장하지 않음", "폭력 장면 없음"

## 관계 변화
| EP | 관계 | 변화 |
|----|------|------|
| 02 | 모치↔마루 | 첫 만남 (경계) |
```

## Phase 2. 에피소드 제작

### 작업 전 자동 검증

에피소드 작업 시작 전 AI가 자동으로 체크:

- [ ] 등장 캐릭터가 characters/에 등록되어 있는가?
- [ ] 배경/장소가 기존 에피소드와 모순되지 않는가? (story-bible 참조)
- [ ] "확정 설정"에 위배되는 내용이 없는가?
- [ ] "금기"에 해당하는 내용이 없는가?
- [ ] 미회수 복선 중 이번에 활용할 것이 있는가? (있으면 제안)

### 스토리 결정 (대화)

1. 사용자가 아이디어 제시 (또는 AI에게 제안 요청)
2. AI가 스토리 초안 제안 (패널별 요약)
3. 사용자 확인/수정
4. 확정 후 다음 단계로

### 에피소드 지시서 작성

**episodes/ep{N}/ep{N}.md**

```markdown
# EP{N} — {제목}

## 메타
- series: {시리즈명}
- 등장인물: [{캐릭터들}]
- 시간대: (story-bible 타임라인과 연결)

## 페이지 1

### 패널 1
- 장소:
- 시간:
- 캐릭터: {이름} — 포즈, 표정
- 배경: 구성 요소
- 소품:
- 말풍선: "대사"
- 효과: (효과음, 시각 효과)

### 패널 2
- ...

### 패널 3
- ...

### 패널 4
- ...

## 페이지 2 (있으면)
- ...
```

### 아스키 프로토타입

지시서 작성 후, 콘솔 스타일 아스키 아트로 레이아웃을 시각화:

- 사이즈는 메타데이터로 별도 기술
- 캐릭터는 이름 + 간단한 표정/포즈로 표현
- 말풍선은 실제 대사 포함
- 효과는 텍스트 설명 또는 기호로
- 패널 구분 명확히

작업 모드가 `careful`이면 사용자 확인 후 진행.
`fast`이면 바로 이미지 생성으로.

## Phase 3. 이미지 생성

### 패널별 API 호출

각 패널마다 개별 호출:

```
입력:
1. 프롬프트 — 해당 패널의 지시서 내용을 SLCT로 변환
2. 스타일 참고 — assets/style-ref.png (매번 동일)
3. 캐릭터 참고 — 해당 패널에 등장하는 캐릭터 ref만
```

> ⚠️ 캐릭터가 여럿이어도, 현재 패널에 등장하는 캐릭터의 참고 이미지만 전송한다.

### 프롬프트 기록

호출한 프롬프트와 결과를 기록:

**episodes/ep{N}/prompts.md**
```markdown
## 패널 1
- prompt: "gray cat sitting at desk, wearing headset..."
- model: flux-klein-4b + webtoon-lora
- result: output/wfh-cat/ep01/panel-01-v1.png
- status: ✅ 채택

## 패널 2
- prompt: "..."
- result: panel-02-v1.png
- status: ❌ 재생성 (표정이 맞지 않음)
- retry-prompt: "... with surprised expression, wide eyes"
- retry-result: panel-02-v2.png
- status: ✅ 채택
```

### 리트라이

- 결과가 불만족 → 사용자 피드백 반영하여 프롬프트 수정 후 재생성
- 기존 버전은 유지 (`-v1`, `-v2` 등)
- 채택된 버전만 최종 합성에 사용

---

## Phase 4. 후처리 (코드)

패널 이미지가 모두 생성되면 코드로 최종 합성:

1. 패널들을 레이아웃에 맞게 배치 (2x2, 세로 등)
2. 말풍선 + 대사 텍스트 오버레이 (만화체 폰트)
3. 효과음 텍스트 배치
4. 나레이션 박스 (있으면)
5. 패널 테두리, gap 적용
6. 최종 사이즈로 출력

출력 위치: `output/{시리즈}/ep{N}/page-{NN}.png`

---

## Phase 5. 완료 후 자동 업데이트

에피소드 완성 후 AI가 story-bible.md 갱신을 제안:

- **타임라인**: 이번 에피소드의 시간/사건 추가
- **확정 설정**: 새로 정해진 설정이 있으면 추가 제안 (사용자 승인 필요)
- **금기**: 이번 에피소드에서 확립된 "안 하는 것" 있으면 제안
- **복선**: 심은 떡밥이 있으면 등록
- **관계 변화**: 캐릭터 간 관계에 변화 있으면 기록
- **새 캐릭터**: 등장했으면 characters/ 추가 제안

## 패널 템플릿

자주 사용하는 구도를 이름으로 정의:

| 템플릿 | 구도 | 적합한 상황 |
|--------|------|-------------|
| establishing | 와이드, 캐릭터 작게, 장소 강조 | 장면 전환, 시작 |
| talk-2char | 좌우 캐릭터, 위에 말풍선 | 대화 장면 |
| talk-3char | 좌-중-우, 말풍선 위 | 3인 대화 |
| reaction-solo | 클로즈업 1인, 배경 단색 | 리액션 컷 |
| action | 동적 포즈, 효과선, 넓은 여백 | 행동/움직임 |
| narration-end | 여백 많음, 나레이션 하단 | 마무리 |
| montage | 3~4 작은 컷 조합 | 시간 경과 |

지시서에서 사용: `template: talk-2char`

## 토큰 시스템

자주 사용하는 요소를 토큰화하여 지시서에서 간결하게 참조:

### 정의 위치

characters/{이름}.md 하단에 `## 토큰` 섹션:

```markdown
## 토큰
- @mochi.sleepy → 졸린 표정, 반쯤 감은 눈, 입 살짝 벌림
- @mochi.headset → 큰 헤드셋 착용
- @bg.room.morning → 모치 원룸, 아침 햇빛, 커튼 사이 빛
```

### 사용

지시서에서:
```markdown
### 캐릭터
- 모치: @mochi.sleepy + @mochi.headset, 중앙, 침대 위
```

## 작업 모드

series.md에서 설정:

| 모드 | 동작 |
|------|------|
| careful | 매 단계(스토리, 프로토타입, 각 패널)에서 사용자 확인 |
| fast | 스토리 확정 후 나머지 자동 실행, 최종 결과만 확인 |

사용자가 대화에서 "빠르게" 또는 "하나씩" 요청하면 해당 에피소드에서만 모드 오버라이드.

## 금지 사항

아래 행동은 어떤 상황에서도 하지 않는다:

- 시리즈 문서(series.md, story-bible.md) 없이 이미지 생성하지 마라
- 캐릭터 시트(characters/{이름}.md) 없이 캐릭터를 그리지 마라 — 시트 먼저 만들 것
- story-bible의 "확정 설정"과 "금기"를 무시하지 마라
- 사용자 확인 없이 스토리를 확정하지 마라
- 프롬프트에 참고 이미지 없이 캐릭터를 생성하지 마라 — 일관성 깨짐
- 에피소드 완료 후 story-bible 갱신 제안을 생략하지 마라
- 작업 전 자동 검증을 건너뛰지 마라

## 각 Phase 완료 조건

### Phase 1 (시리즈 셋업) 완료 조건
- [ ] series.md가 생성되고 기본 설정이 모두 채워져 있음
- [ ] 최소 1명의 캐릭터가 characters/에 등록됨 (시트 + 참고 이미지)
- [ ] 그림체 참고 이미지가 assets/style-ref에 저장됨
- [ ] story-bible.md 템플릿이 생성됨
- [ ] 사용자가 셋업 내용을 확인함

### Phase 2 (에피소드 제작) 완료 조건
- [ ] 사전 검증 통과 (설정 모순 없음)
- [ ] 에피소드 지시서가 모든 패널에 대해 작성됨
- [ ] 사용자가 스토리를 승인함
- [ ] 아스키 프로토타입이 생성됨 (careful 모드 시 사용자 확인까지)

### Phase 3 (이미지 생성) 완료 조건
- [ ] 모든 패널 이미지가 생성됨
- [ ] 리트라이한 경우, 채택 버전이 결정됨
- [ ] prompts.md에 사용한 프롬프트가 기록됨

### Phase 4 (후처리) 완료 조건
- [ ] 최종 합성 이미지가 output/{시리즈}/ep{N}/에 저장됨
- [ ] 사이즈, 포맷이 series.md 설정과 일치함

### Phase 5 (자동 업데이트) 완료 조건
- [ ] story-bible.md 갱신 항목이 사용자에게 제안됨
- [ ] 사용자 승인된 항목이 반영됨

---

## 원칙: 단계 생략 금지

어떤 상황에서도 아래 순서를 뛰어넘지 않는다:

1. 시리즈 문서 존재 확인 → 없으면 셋업부터
2. 스토리 확정 → 사용자 승인 필수
3. 사전 검증 → 통과해야 다음 진행
4. 이미지 생성 → 도구 세팅 확인 필수
5. 완료 후 → story-bible 갱신 제안 필수

"빨리 해줘"라는 요청이 와도, fast 모드로 전환할 뿐 단계 자체를 건너뛰지 않는다.
fast 모드는 "사용자 확인을 줄이는 것"이지 "단계를 생략하는 것"이 아니다.

## 연재 중 변경 관리

시리즈 연재 도중 캐릭터 추가, 그림체 변경, 설정 변경이 필요할 때의 프로세스.

### 신규 캐릭터 추가

1. 사용자가 새 캐릭터 정보 제공 (이름, 성격, 외형 설명 등)
2. `characters/{이름}.md` 시트 작성
3. **캐릭터 참고 이미지 생성**:
   - 사용자가 참고 이미지를 직접 제공하면 그것을 사용
   - 없으면 → 외형 설명 + 기존 style-ref 기반으로 이미지 생성 API 호출
   - 프롬프트: 캐릭터 외형 설명 + "character sheet, front view, neutral expression" + 시리즈 그림체
   - 생성된 이미지를 사용자에게 확인 ("이 캐릭터 맞아요?")
   - 승인 후 `assets/characters/{이름}-ref.png`로 저장
4. `characters/_index.md`에 추가
5. story-bible.md에 첫 등장 에피소드 기록
6. **(중요)** 기존 에피소드와 모순 없는지 확인
   - 이전에 "등장인물 없음" 설정이면 등장 이유 명시 필요
   - 기존 캐릭터와의 관계 정의

### 그림체 변경

시리즈 도중 그림체를 바꾸고 싶을 때:

1. 새 스타일 참고 이미지 제공
2. `assets/style-ref.png`를 교체하지 않음 → `assets/style-ref-v2.png`로 추가
3. `series.md`에 변경 기록:
   ```markdown
   ## 그림체 변경 이력
   | 적용 EP | 참고 이미지 | 변경 이유 |
   |---------|------------|----------|
   | ep01~07 | style-ref-v1.png | 초기 스타일 |
   | ep08~ | style-ref-v2.png | 선 굵기 줄임, 색감 밝게 |
   ```
4. 변경 에피소드부터 새 참고 이미지 사용
5. 이전 에피소드의 캐릭터 ref 이미지도 새 스타일로 갱신할지 사용자에게 확인

### 설정 변경 (확정 사항 수정)

story-bible의 "확정 설정"을 바꿔야 할 때:

1. 사용자가 변경 요청
2. AI가 영향 분석:
   - 이 설정이 사용된 기존 에피소드 목록
   - 변경 시 모순이 생기는 부분
3. 사용자 확인 후 변경 적용
4. story-bible.md에 변경 이력 기록:
   ```markdown
   ## 설정 변경 이력
   | EP | 변경 전 | 변경 후 | 이유 |
   |----|---------|---------|------|
   | ep08 | 모치 원룸 | 모치 투룸으로 이사 | 스토리 전개 |
   ```

### LoRA/모델 변경

이미지 생성 모델이나 LoRA를 바꿀 때:

1. 새 설정을 series.md에 기록
2. 테스트 이미지 1장 생성 → 기존 에피소드와 스타일 차이 확인
3. 차이가 크면 사용자에게 경고 ("ep07까지와 톤이 달라질 수 있습니다")
4. 승인 후 적용

### 변경 관리 원칙

- 모든 변경은 **기록**한다 (언제, 뭘, 왜)
- 기존 파일을 **덮어쓰지 않고** 버전을 추가한다
- 변경 전 **영향 분석**을 한다 (기존 에피소드와 모순 체크)
- 사용자 **승인 없이** 설정을 변경하지 않는다
