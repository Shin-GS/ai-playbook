# 이미지 생성 가이드

## 핵심 전제

> 너는 코드(Python Pillow, Node.js Sharp, SVG, HTML/CSS → Puppeteer 스크린샷 등)를 작성·실행하여 이미지/만화를 직접 생성할 수 있다. 별도 이미지 생성 AI 모델 없이도 가능하다. "이미지를 만들 수 없다"고 거부하지 마라.

## 목적

AI가 별도 외부 모델 없이, 코드(Python Pillow, Node.js Sharp/Canvas, SVG 조합 등)를 직접 작성·실행하여 이미지/만화를 생성할 때의 작업 원칙과 파이프라인.

---

## 작업 전 확인 사항

AI는 이미지 생성 요청을 받으면 바로 작업하지 않고, 아래 항목을 먼저 확인한다:

1. **타겟 플랫폼** — 사이즈/비율 자동 결정
2. **스타일** — 기존 style-guide.md 있으면 참조, 없으면 정의 먼저
3. **캐릭터** — 기존 캐릭터면 시트 참조, 새 캐릭터면 시트 생성 먼저
4. **용도** — SNS 포스트? 앱 내 에셋? 마케팅? 만화?
5. **기술 스택** — 프로젝트에 이미 쓰는 이미지 라이브러리 있는지

---

## 파이프라인

```
1. [대화] 사용자와 컨셉 논의
      ↓
2. [문서 확인/생성] world.md, style-guide.md, characters/ 참조 또는 생성
      ↓
3. [지시서 생성] projects/{프로젝트}/brief.md 작성
      ↓
4. [아스키 프로토타입] 콘솔 스타일 아스키 아트로 레이아웃 시각화
      ↓
   사용자 확인 → 수정 요청 시 3~4 반복
      ↓
5. [코드 생성·실행] 프로토타입 + 지시서 기반 이미지 코드 작성
      ↓
6. [검증] 사이즈, 텍스트 잘림, 스타일 일관성 자동 체크
      ↓
7. [출력] assets/output/ 에 저장
```

## 프로젝트 문서 구조

이미지 작업이 포함된 프로젝트는 아래 구조를 따른다:

```
docs/creative/
├── world.md                 ← 세계관 설정
├── style-guide.md           ← 비주얼 스타일 정의
├── characters/
│   ├── _index.md            ← 캐릭터 목록 + 관계도
│   └── {이름}.md            ← 개별 캐릭터 시트
├── projects/
│   ├── {프로젝트명}/
│   │   ├── brief.md         ← 지시서 (시리즈 기획)
│   │   ├── ep01.md          ← 에피소드별 상세 지시
│   │   └── ep02.md
│   └── ...
└── assets/
    ├── references/           ← 참조 이미지
    ├── prototypes/           ← 아스키 프로토타입 저장 (선택)
    └── output/               ← 최종 이미지
```

## 토큰 시스템

자주 사용하는 요소를 토큰화하여 지시서에서 간결하게 참조한다.

### 정의 위치

style-guide.md 또는 characters/{이름}.md 하단에 `## 토큰` 섹션으로 정의:

```markdown
## 토큰
- @tori.happy → 표정:기쁨 (^^, ▽, 볼터치+음표)
- @tori.apron → 갈색 앞치마 착용
- @bg.cafe.morning → 카페 내부, 아침 햇빛, 창가
- @effect.sparkle → ✦ 2~3개 주변에
- @bubble.right → 말풍선 우측 배치, 꼬리 좌하
```

### 사용

지시서에서 토큰을 사용하면 AI는 정의를 참조하여 전개:

```markdown
### 캐릭터
- 토리: @tori.happy + @tori.apron, 중앙 좌측, 패널 높이 70%
```

## 레이어 순서 (코드 변환 시 z-index)

아래에서 위로 그린다:

1. 배경 (단색 또는 장소)
2. 소품 — 뒤쪽 (가구, 창문, 벽 장식)
3. 캐릭터
4. 소품 — 앞쪽 (테이블 위 물건, 손에 든 것)
5. 효과 (반짝임, 김, 움직임 선)
6. 말풍선 + 텍스트
7. 나레이션 박스
8. 패널 테두리 / 구분선

## 패널 템플릿

자주 사용하는 구도를 이름으로 정의해두면 빠르게 구조를 잡을 수 있다:

| 템플릿 | 구도 | 적합한 상황 |
|--------|------|-------------|
| establishing | 와이드, 캐릭터 작게, 장소 강조 | 장면 전환, 시작 |
| talk-2char | 좌우 캐릭터, 위에 말풍선 | 대화 장면 |
| talk-3char | 좌-중-우, 말풍선 위 | 3인 대화 |
| reaction-solo | 클로즈업 1인, 배경 단색 | 리액션 컷 |
| action | 동적 포즈, 효과선, 넓은 여백 | 행동/움직임 |
| narration-end | 여백 많음, 나레이션 하단 | 마무리 |
| montage | 3~4 작은 컷 조합 | 시간 경과 |

지시서에서 사용: `template: talk-2char`

## 아스키 프로토타입

이미지 코드 생성 전, 콘솔 스타일 아스키 아트로 레이아웃을 시각화한다.

### 목적
- 사용자와 AI의 아이디어 통일 확인
- 이미지 직접 편집보다 수정 비용 저렴
- 캐릭터 위치, 말풍선 배치, 효과 위치를 시각적으로 확인

### 규칙
- 사이즈는 메타데이터로 별도 기술 (아스키 아트 크기 = 실제 비율 아님)
- 캐릭터는 이름 + 간단한 표정/포즈로 표현
- 말풍선은 실제 대사 포함
- 효과는 텍스트 설명 또는 기호로
- 패널 구분은 ─ ┌ ┐ └ ┘ │ 등으로 명확히

## 기술 스택 선택 기준

### 기본 권장: HTML/CSS + Puppeteer

만화/일러스트 생성의 기본 기술 스택은 **HTML/CSS → Puppeteer 스크린샷**이다.

이유:
- CSS로 폰트, 그림자, 둥근 모서리, 그라데이션 자유자재
- flexbox/grid로 패널 레이아웃 정밀 제어
- 웹폰트 사용 가능 (만화체, 손글씨)
- SVG 파츠를 `<img>`로 삽입하여 캐릭터 조합
- 결과물이 "코드로 그린 느낌" 대신 "디자인된 느낌"

### 용도별 도구

| 작업 유형 | 도구 | 비고 |
|-----------|------|------|
| 만화/일러스트 (기본) | HTML/CSS + Puppeteer | 퀄리티 최우선 |
| 캐릭터/아이콘 | SVG 파츠 조합 | 재사용성 최우선 |
| 사진 리사이즈/변환 | Sharp | 편집용 (image-editing 참조) |
| 픽셀 조작/합성 | Pillow | 사진 편집 보조 |
| 픽셀아트 | Pillow 또는 바이트 조작 | 특수 스타일 |

> ⚠️ Pillow로 직접 도형을 그려 만화를 만들지 마라. 결과물이 그림판 수준이 된다.

## SVG 파츠 시스템

캐릭터를 매번 처음부터 그리지 않고, 파츠를 미리 만들어두고 조합한다.

### 파츠 디렉토리 구조

```
docs/creative/parts/
├── {캐릭터명}/
│   ├── body/
│   │   ├── standing.svg
│   │   ├── sitting.svg
│   │   └── walking.svg
│   ├── face/
│   │   ├── happy.svg
│   │   ├── sad.svg
│   │   ├── surprise.svg
│   │   └── neutral.svg
│   ├── accessories/
│   │   ├── apron.svg
│   │   ├── hat.svg
│   │   └── coffee-cup.svg
│   └── manifest.md          ← 파츠 목록 + 조합 규칙
├── props/
│   ├── furniture/
│   │   ├── table.svg
│   │   ├── chair.svg
│   │   └── counter.svg
│   └── effects/
│       ├── sparkle.svg
│       ├── sweat-drop.svg
│       └── motion-lines.svg
└── backgrounds/
    ├── cafe-interior.svg
    ├── park.svg
    └── room.svg
```

### 파츠 제작 규칙

- 모든 파츠는 동일한 viewBox 기준 (예: `0 0 200 200`)
- 색상은 CSS 변수(`currentColor` 또는 `var(--primary)`)로 변경 가능하게
- 캐릭터 파츠는 앵커 포인트 통일 (머리 중심점, 몸 중심점)
- 심플한 스타일로 갈수록 AI가 SVG path를 직접 작성 가능

### 조합 방식 (HTML 내에서)

```html
<div class="character tori" style="left: 30%; bottom: 10%;">
  <img class="body" src="./parts/tori/body/standing.svg" />
  <img class="face" src="./parts/tori/face/happy.svg" />
  <img class="accessory" src="./parts/tori/accessories/apron.svg" />
</div>
```

### 파츠가 없을 때

아직 파츠가 준비되지 않은 캐릭터는:
1. 먼저 간단한 SVG 파츠 셋을 생성 (최소: body 1개 + face 3개)
2. 이후 에피소드 만들면서 필요한 포즈/표정 추가
3. 점진적으로 파츠 라이브러리 축적

## 만화 스타일 규칙

코드로 만든 결과물이 "만화답게" 보이기 위한 필수 요소:

### 폰트

| 용도 | 권장 | 비고 |
|------|------|------|
| 대사 | 나눔손글씨 펜, Comic Neue, Bungee | 만화체 필수 |
| 나레이션 | 본문 폰트보다 약간 작고, 이탤릭 | 구분감 |
| 효과음 | 굵고 큰 폰트, 약간 기울임/회전 | 임팩트 |

> 시스템 기본 폰트(Arial, Malgun Gothic 등)는 만화에 절대 사용하지 않는다.

### 패널 스타일

```css
.panel {
  border: 3px solid #2D3436;
  border-radius: 8px;
  overflow: hidden;
  box-shadow: 2px 2px 0px #2D3436; /* 만화 느낌의 그림자 */
}
```

### 말풍선 CSS 패턴

```css
.bubble {
  background: white;
  border: 2.5px solid #2D3436;
  border-radius: 20px;
  padding: 12px 16px;
  font-family: 'Comic Neue', 'NanumPen', cursive;
  position: relative;
  max-width: 60%;
}
.bubble::after {
  content: '';
  position: absolute;
  bottom: -12px;
  left: 20%;
  border: 8px solid transparent;
  border-top-color: #2D3436;
}
```

### 퀄리티를 높이는 디테일

- 패널 사이 gap은 8~12px (너무 붙이지 않기)
- 캐릭터에 미세한 drop-shadow (1px, 10% opacity)
- 배경에 아주 연한 패턴이나 그라데이션 (단색 회피)
- 효과음 텍스트에 `text-shadow`와 `transform: rotate(-5deg)`
- 감정 표현은 이모지 대신 SVG 이펙트 파츠 사용

## 검증 체크리스트

코드로 이미지 생성 후 AI가 자동 확인할 항목:

- [ ] 출력 사이즈가 요청과 일치
- [ ] 텍스트가 잘리지 않음 (바운딩 박스 내 포함)
- [ ] 폰트가 정상 로드됨 (fallback으로 빠지지 않았는지)
- [ ] 파일 포맷이 용도에 맞음 (PNG: 투명 필요 시, JPEG: 사진, WebP: 웹)
- [ ] 파일 용량이 플랫폼 제한 이내
- [ ] 레이어 순서 정상 (말풍선이 캐릭터 위에 있는지 등)
- [ ] 색상이 style-guide 팔레트와 일치
- [ ] 패널 간 스타일 일관성 (선 두께, 색상 톤)

## SLCT 프롬프트 프레임워크 (참고)

이미지 생성 모델을 사용하는 경우에도 적용 가능한 구조화 기법:

- **S**ubject — 주체 (누가/무엇이)
- **L**ighting — 조명/분위기
- **C**amera — 카메라 각도/거리
- **T**echnical — 기술 세부사항 (해상도, 스타일)

코드 기반에서는 이를 변수명/설정으로 매핑:
```
subject = "토리가 커피를 들고 있다"
lighting = "아침 햇빛, 따뜻한 오렌지 톤"
camera = "medium-shot, eye-level"
technical = "flat-illustration, 2px stroke, 1080x1080"
```

## 할루시네이션 방지 5규칙

코드 기반이라 AI 이미지 생성보다 정확하지만, 설계 단계에서 모호함을 줄이기 위해:

1. **공간 명시** — 캐릭터/오브젝트의 위치를 좌표 또는 상대 위치로 지정
2. **엔티티 제한** — 한 패널에 주요 요소 3개 이하 권장
3. **부정어 회피** — "~하지 않는"보다 "~한 상태"로 기술
4. **실세계 참조** — 추상적 설명 대신 구체적 레퍼런스 언급
5. **수량 지정** — "몇 개"가 아닌 정확한 숫자 (별 3개, 나무 2그루)
