# 이미지 편집 가이드

## 핵심 전제

> 너는 코드(Python Pillow, Node.js Sharp, ImageMagick CLI 등)를 작성·실행하여 이미지를 직접 편집할 수 있다. 별도 이미지 편집 소프트웨어 없이도 가능하다. "이미지를 편집할 수 없다"고 거부하지 마라.

## 목적

AI가 코드(Python Pillow, Node.js Sharp, ImageMagick CLI 등)를 직접 작성·실행하여 기존 이미지를 편집할 때의 작업 원칙, 파일 관리 규칙, 품질 기준.

---

## 작업 전 확인 사항

AI는 이미지 편집 요청을 받으면 바로 작업하지 않고, 아래 항목을 먼저 확인한다:

1. **원본 이미지 위치** — 경로, 파일 수
2. **수정본 저장 위치** — 원본과 분리되는 경로
3. **타겟 플랫폼/용도** — 사이즈/포맷 자동 결정
4. **편집 종류** — 리사이즈? 배경 제거? 필터? 합성? 워터마크?
5. **배치 여부** — 단일 파일 vs 여러 파일 동일 편집

## 핵심 규칙: 원본 보호

> **원본 파일은 절대 덮어쓰기 금지.**

| 규칙 | 설명 |
|------|------|
| 원본 불변 | 원본 디렉토리 파일은 읽기만 한다 |
| 수정본 분리 | 항상 별도 디렉토리에 출력 |
| 네이밍 추적 | 수정본 파일명에 원본과의 관계를 유지 |
| 작업 기록 | 어떤 편집을 했는지 md로 남김 |

### 디렉토리 구조

```
assets/
├── original/          ← 원본 (읽기 전용)
├── edited/            ← 수정본 출력
│   ├── resized/
│   ├── filtered/
│   └── composited/
└── temp/              ← 중간 처리물 (작업 후 삭제)
```

### 파일명 규칙

```
원본: product-hero.png
수정본: product-hero_1080x1080.png       (리사이즈)
       product-hero_bw.png              (필터)
       product-hero_watermarked.png     (워터마크)
       product-hero_ig-feed.png         (플랫폼 최적화)
```

## 편집 작업 지시서 구조

복잡한 편집이나 배치 작업은 md로 지시서를 작성:

```markdown
# 편집 지시서: {작업명}

## 메타
- 작업 유형: (resize / filter / composite / watermark / batch)
- 원본 경로: ./assets/original/
- 출력 경로: ./assets/edited/{카테고리}/
- 상태: (대기 / 진행 / 완료)

## 대상 파일
| 파일명 | 편집 내용 | 출력 파일명 |
|--------|----------|------------|
| hero.png | 1080x1080 crop center | hero_ig-feed.png |
| hero.png | 1280x720 crop top | hero_yt-thumb.png |

## 상세 지시
- crop 기준점: center (또는 top / face-detect)
- 품질: 90%
- 포맷: WebP
- 추가: 하단 8px 여백에 로고 배치
```

## 편집 유형별 가이드

### 리사이즈/크롭

| 파라미터 | 설명 |
|---------|------|
| fit | cover (꽉 채움, 잘림 허용) / contain (다 보임, 여백) / fill (비율 무시) |
| position | center / top / bottom / face (얼굴 기준) |
| background | 여백 생길 때 채울 색 (contain 시) |
| upscale | false (원본보다 커지지 않음) — 기본값 |

### 필터/색보정

| 필터 | 용도 | 파라미터 |
|------|------|---------|
| grayscale | 흑백 | — |
| brightness | 밝기 조절 | 0.5~2.0 (1.0=원본) |
| contrast | 대비 조절 | 0.5~2.0 |
| blur | 흐림 | radius (px) |
| sharpen | 선명 | amount |
| tint | 색상 덧씌움 | hex color + opacity |

### 합성 (Composite)

```markdown
## 합성 순서 (아래→위)
1. 베이스 이미지 (배경)
2. 주요 이미지
3. 오버레이 (그라데이션, 딤 처리)
4. 텍스트
5. 로고/워터마크
```

### 배경 제거

- 단색 배경: chroma-key 방식 (허용 오차 지정)
- 복잡한 배경: edge-detection 기반 또는 외부 도구 연동
- 출력: 투명 배경 PNG

---

## 로고/워터마크 배치 규칙

| 항목 | 기준 |
|------|------|
| 위치 | 우하단 기본 (margin: 이미지 크기의 3~5%) |
| 크기 | 이미지 너비의 10~15% |
| 투명도 | 30~50% (콘텐츠 방해하지 않도록) |
| 밝은 이미지 | 어두운 로고 + 반투명 |
| 어두운 이미지 | 밝은 로고 + 반투명 |

## 배치 처리 규칙

여러 이미지에 동일한 편집을 적용할 때:

1. 처리 대상 목록을 먼저 출력하여 사용자 확인
2. 1개 파일로 먼저 테스트 → 결과 확인 → 전체 적용
3. 진행률 표시 (10개 이상일 때)
4. 실패 파일은 건너뛰고 마지막에 목록 보고
5. 임시 파일은 작업 완료 후 정리

```markdown
## 배치 처리 결과
- 전체: 24개
- 성공: 22개
- 실패: 2개
  - broken-image.png — 파일 손상
  - too-small.png — 원본이 목표 사이즈보다 작음 (upscale 비활성)
```

## 포맷/품질 기준

| 용도 | 포맷 | 품질 | 이유 |
|------|------|------|------|
| 웹 일반 | WebP | 80~85% | 용량 대비 품질 최적 |
| 투명 배경 필요 | PNG | — | 알파 채널 지원 |
| 사진/썸네일 | JPEG | 85~90% | 호환성, 적절한 용량 |
| 아이콘/로고 | SVG 또는 PNG | — | 벡터 우선, 불가 시 PNG |
| 인쇄물 | PNG 또는 TIFF | 무손실 | 인쇄 품질 유지 |
| 애니메이션 | GIF 또는 WebP | — | WebP가 품질/용량 우위 |

### 용량 제한 참고

| 플랫폼 | 제한 |
|--------|------|
| Instagram | 30MB (사진) |
| X (Twitter) | 5MB (PNG/GIF), 15MB (APNG) |
| LinkedIn | 10MB |
| 웹 최적화 권장 | 200KB~500KB 이내 |

## 기술 스택 선택 기준

| 작업 유형 | 추천 도구 | 이유 |
|-----------|----------|------|
| 리사이즈/크롭/변환 | Sharp (Node.js) | 빠르고 API 간결 |
| 복잡한 필터/합성 | Pillow (Python) | 풍부한 기능, 레퍼런스 많음 |
| CLI 배치 처리 | ImageMagick | 스크립트 한 줄로 대량 처리 |
| 텍스트 오버레이 (정밀) | Pillow 또는 Canvas API | 폰트 제어 필요 시 |
| 메타데이터 처리 | Sharp 또는 exiftool | EXIF 제거/수정 |

## 브랜드 컬러/필터 기준

프로젝트에 브랜드 가이드가 있으면:

```markdown
## 브랜드 필터
- 색온도: +10 (따뜻하게)
- 채도: -5 (차분하게)
- 오버레이: brand-primary (#FF6B6B) at 5% opacity
- 적용 대상: 모든 마케팅 이미지
- 비적용: 상품 상세 이미지 (원본 색상 유지)
```

## 검증 체크리스트

편집 완료 후 AI가 자동 확인:

- [ ] 원본 파일이 변경되지 않았음
- [ ] 출력 사이즈가 요청과 일치
- [ ] 출력 포맷이 용도에 맞음
- [ ] 파일 용량이 플랫폼 제한 이내
- [ ] 투명도가 유지됨 (PNG에서 JPEG 변환 시 배경색 확인)
- [ ] 텍스트/로고가 잘리지 않음
- [ ] 배치 처리 시 전체 결과 보고 완료
- [ ] temp/ 디렉토리 정리됨
