# 서브에이전트 협업 워크플로우

## 코드 수정 시 서브에이전트 활용 원칙

코드 수정이 필요한 작업에서는 서브에이전트가 직접 코드를 수정하도록 위임한다.
main 에이전트는 분석/조율/결과 보고만 담당한다.

> 에이전트 목록은 프로젝트별 정의

## Steps

### Phase 1: 기획

- **done-when**: 기획 문서 작성 완료, 기존 문서와 충돌 없음 확인
- **fail-action**: 기존 문서와 모순 발견 시 충돌 내용 보고 + 해결 방안 제안

절차:
1. 기획 에이전트 호출
2. 기존 문서 확인 → 케이스 정의 작성/수정
3. 필요 시 비즈니스 로직/용어 문서 업데이트
4. 다음 단계 전달 사항 명시

### Phase 2: 디자인

- **done-when**: 디자인 명세 생성 완료, 토큰 변경 시 영향 범위 보고 완료
- **fail-action**: 케이스 해석 불명확 시 기획자에게 재확인 요청

절차:
1. 디자인 에이전트 호출 (기획 산출물 전달)
2. 케이스 기반 디자인 명세 구현
3. 토큰/컴포넌트 변경 시 영향 범위 + FE 동기화 필요 여부 보고

### Phase 3: FE/BE 구현

- **done-when**: 빌드 통과 + 관련 테스트 통과
- **fail-action**: 2회 실패 시 접근 방식 변경, 사용자 알림

절차:
1. 서브에이전트 병렬 호출 (FE/BE)
2. 디자인 명세 기반 FE 구현
3. API 구현

### Phase 4: 리뷰

- **done-when**: 코드 리뷰 피드백 처리 완료 (1회 리뷰 원칙)
- **fail-action**: P0 이슈 발견 시 해당 developer에게 수정 지시

절차:
1. 코드 리뷰 서브에이전트 호출 (병렬 가능)
2. main 에이전트: 리뷰 피드백 평가 후 수정 필요 여부 판단
3. 아키텍처 이슈 검증 (필요 시)

### Phase 5: 보고

- **done-when**: 사용자에게 변경 내용 + 영향 범위 보고 완료
- **fail-action**: 없음 (최종 단계)

## 연쇄 호출 판단 기준

main 에이전트가 아래 기준으로 다음 에이전트 호출 여부를 판단한다.

### 기획 → 디자인

| 기획자 산출물 | 디자이너 호출 필요? |
|--------------|-------------------|
| 새 케이스 추가 | Yes |
| 기존 케이스 삭제/변경 | Yes |
| 새 화면 추가 | Yes |
| 비즈니스 로직만 변경 (UI 무관) | No |
| 용어만 변경 | No (단, UI 텍스트 변경이면 Yes) |

### 디자인 → FE 개발

| 디자이너 산출물 | FE 개발자 호출 필요? |
|----------------|---------------------|
| 디자인 토큰 변경 | Yes — 설정 동기화 |
| 컴포넌트 스타일 변경 | 상황에 따라 (FE에 해당 컴포넌트 있으면 Yes) |
| HTML 명세만 변경 (토큰 무변경) | No (FE 구현 요청이 별도로 있을 때만) |

## 서브에이전트 호출 시 주의사항

- 프롬프트에 "직접 파일을 수정해줘"를 명시할 것
- 분석/계획만 요청하지 말고, 수정까지 완료하도록 지시할 것
- 변경 대상 파일 경로와 현재 코드 컨텍스트를 충분히 제공할 것
- FE/BE 간 인터페이스(API 스펙, DTO 구조)가 변경되는 경우 양쪽에 동일한 스펙을 전달할 것
- contextFiles로 경로만 전달, 인라인 코드 블록 금지

## 코드 리뷰 호출 시 주의사항

- 리뷰 대상 파일 경로와 변경 내용을 충분히 제공할 것
- 리뷰어는 코드 수정을 직접 하지 않고, 피드백만 반환할 것
- 피드백 기반 수정 후 재리뷰는 하지 않는다 (1회 리뷰 원칙, 무한 루프 방지)

## main 에이전트가 직접 수정해도 되는 경우

- steering/spec 등 설정 파일 수정
- 단일 파일의 간단한 수정 (한두 줄 변경)
- 스타일 변경, 텍스트 수정 등 단순 수정
- 문서 단순 업데이트

## 검증 (Phase 3 완료 시)

코드 구현 완료 후 반드시 `verification-loop` 워크플로우를 적용한다:
1. 빌드 → 타입 체크 → 린트 → 테스트 순서로 검증
2. 실패 시 루프 규칙에 따라 수정→재검증
3. 모든 단계 통과 후 Phase 4(리뷰)로 진행

## QA 연쇄 (기능 구현 완료 시)

코드 변경 후 아래 기준으로 QA 업데이트 필요 여부를 판단한다:

### QA 업데이트 필요
- 사용자 행동에 영향을 주는 변경 (새 기능, UI 변경, API 변경)
- 비즈니스 규칙 변경 (계산 로직, 상태 전이, 제약 조건)
- 에러 처리 방식 변경

### QA 업데이트 불필요
- 내부 리팩토링 (동작 변경 없음)
- 성능 개선 (사용자 동작 동일)
- 코드 포맷팅/린트 수정
- 문서/주석 변경

### 절차
1. `verification-loop` 통과 (빌드/타입/린트/테스트)
2. QA 업데이트 필요 판단
3. 필요하면: `qa-tester` 에이전트에게 영향 분석 요청
   - 기존 테스트 중 영향받는 항목 수정
   - 새 케이스 추가 필요 시 추가
4. 불필요하면: 바로 Phase 5(보고)로

## 기획/디자인 결정 시

기획 변경이나 디자인 결정이 필요할 때 `multi-perspective-review` 워크플로우를 적용한다:
- 단순 구현이 아닌, 방향 결정이 필요한 상황에서만
- 보수적/사용자/비판적/창의적 4관점 분석 후 종합

## Git 커밋 규칙

- `git add -A` 또는 `git add .` 사용 금지
- 반드시 변경 대상 파일을 명시적으로 `git add <파일1> <파일2> ...`로 스테이징
- 커밋 전 `git diff --cached --stat`으로 스테이징 파일 확인
- untracked 파일은 사용자가 명시적으로 요청한 경우에만 커밋에 포함
