# 검증 루프 (Verification Loop)

## 목적

코드 변경 후 "작동하는 상태"를 보장하기 위한 검증 사이클. 실패 시 자동 수정→재검증을 반복하되, 무한 루프를 방지한다.

## 검증 단계 (프로젝트에 맞게 선택)

1. **빌드** — 컴파일 통과
2. **타입 체크** — 타입 에러 없음
3. **린트** — 코드 품질 규칙 위반 없음
4. **테스트** — 관련 테스트 통과
5. **보안 스캔** — (선택) 명백한 보안 이슈 없음
6. **문서 동기화** — (선택) 코드 변경이 문서에 영향을 주는지 확인

## Steps

### Step 1: 변경 완료 후 검증 실행
- **done-when**: 모든 검증 단계 통과
- **fail-action**: 실패한 단계의 에러 분석 → Step 2로

### Step 2: 에러 수정
- **done-when**: 에러 원인 파악 + 수정 완료
- **fail-action**: 원인 파악 불가 시 사용자에게 에러 보고

### Step 3: 재검증
- **done-when**: 실패했던 단계 재실행 → 통과
- **fail-action**: 같은 에러 반복 시 Step 4로

### Step 4: 접근 방식 변경
- **done-when**: 다른 방식으로 수정하여 통과
- **fail-action**: 3회 반복 실패 시 사용자에게 보고 + 대안 제시

## 루프 규칙

| 상황 | 행동 |
|------|------|
| 1회 실패 | 해당 에러만 수정 → 해당 단계만 재실행 |
| 같은 에러 2회 반복 | 접근 방식 변경 (다른 방법 시도) |
| 같은 에러 3회 반복 | 즉시 중단 + 사용자에게 보고 + 시도한 것/안 된 이유/대안 제시 |
| 수정이 다른 단계를 깨뜨림 | 전체 검증 단계 처음부터 재실행 |
| 5분 이상 같은 에러에 갇힘 | 중단 + 보고 |

### Step 6: 문서 동기화 확인 (선택)
- **done-when**: 코드 변경과 관련된 프로젝트 문서가 모두 최신 상태
- **fail-action**: 갱신 필요한 문서 목록을 사용자에게 안내

> 이 단계는 `project-docs-structure`의 "문서 ↔ 코드 매핑" 테이블을 기준으로 판단한다.
> 내부 리팩토링 등 문서 영향 없는 변경이면 스킵.

## 검증 완료 보고

검증 통과 시 짧게 보고:
```
✅ 검증 완료
- 빌드: 통과
- 타입 체크: 통과
- 린트: 통과 (경고 2건 — 기존 코드)
- 테스트: 12/12 통과
```
